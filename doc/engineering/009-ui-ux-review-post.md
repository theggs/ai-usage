## 新发现的问题

### 25.【P1】面板底部卡片被截断，缺少底部间距
**截图实证**：面板截图中 Codex 卡片被窗口底部截断，最后一张卡片紧贴滚动容器边缘，无呼吸空间。
**问题**：用户滚动到最底部时，最后一张卡片紧贴容器底边，视觉上缺乏完成感，且容易误以为下方还有被遮挡的内容。
**建议**：在 PanelView 或滚动容器底部添加 `pb-4` 间距。

### 26.【P2】设置页分组标题与卡片间距过大
**截图实证**：设置截图中"通用"标题和第一张卡片（语言）之间、"显示"标题和前一张卡片（刷新间隔）之间，间距视觉上偏大。去掉 PreferenceSection 的卡片包裹后，组标题和内容之间缺少视觉归属感。
**问题**：去卡片化后，分组标题和下方内容的从属关系不够紧密。用户可能不确定标题是属于上方还是下方。
**建议**：
- 组标题上方间距大于下方间距（如上 `mt-5`、下 `mb-1`），拉近标题与其内容的视觉距离
- 或在组标题下加一条细分隔线

### 27.【P2】代理"应用"按钮视觉权重过高
**代码实证**：`SettingsView.tsx:269` 代理应用按钮使用 `bg-emerald-50 border-emerald-200 text-emerald-900 font-semibold`，是设置页中唯一带颜色填充的按钮。
**问题**：代理设置是低频操作，但其"应用"按钮的视觉权重高于所有即时生效的设置项，造成注意力吸引不合理。与 FR-019 对通知测试按钮的降权处理不一致。
**建议**：降为 `border-slate-200 bg-white text-slate-700`（与通知测试按钮同级），或仅在代理配置有未保存更改时才高亮。同时将按钮标签从"应用代理"/"Apply proxy"简化为"应用"/"Apply"——在代理设置上下文中含义已足够明确。

### 28.【P1】面板与设置 header 设计不统一
**截图实证**：面板 header 左侧有两行信息（副标题+摘要），右侧两个圆形图标按钮（刷新+设置）；设置 header 左侧仅一行小字"设置"，右侧"< 返回"文字胶囊按钮。
**问题**：
1. 两个 header 的信息密度和视觉重心差异大。设置 header 左侧空间利用率低，显得空旷。
2. "额度面板"副标题和"设置"标题在两个视图中的垂直位置不对齐（面板是两行结构，设置是单行）。
3. 设置页"< 返回"是文字胶囊按钮（`px-3 py-1`），与面板侧的圆形图标按钮（`h-9 w-9`）风格不一致。
**建议**：
- 返回按钮改为与面板侧一致的圆形图标按钮（只保留 BackIcon chevron，去掉文字）
- 设置 header 左侧可在"设置"下方加一行保存状态反馈（"已保存"/"保存中..."），对齐面板 header 的两行信息密度

### 29.【P1】警示卡片头部大片空白
**截图实证**：Claude Code 卡片（warning 状态）在服务名"Claude Code"上方有约 56px 的空白区域。
**代码实证**：`ServiceCard.tsx:35-37`，accent bar 使用 `h-10 w-1 -ml-4 mb-4`（高度 40px + 下方 margin 16px），作为块级元素占据文档流，将服务名推到了很下方。
**问题**：accent bar 的本意是卡片左侧的视觉强调（类似 Material Design 的 left border），但当前实现让它占据了一行独立空间，而非沿卡片左边缘纵向延伸。正常状态卡片无此问题（已修复为不渲染），但 warning/danger 卡片仍受影响。
**建议**：将 accent bar 改为绝对定位（`absolute left-0 top-0 h-full w-1`），不占文档流，沿卡片左边缘全高显示。

### 30.【P1】设置图标绿色边框无语义含义
**截图实证**：面板 header 右侧两个按钮中，刷新按钮使用 `border-slate-200 text-slate-700`（中性灰），设置按钮使用 `border-emerald-200 text-emerald-700`（绿色）。
**代码实证**：`AppShell.tsx:272`，设置按钮硬编码了 emerald 色系。
**问题**：两个同级功能按钮使用不同色系，用户会误以为绿色有特殊含义（如"已保存"、"正常状态"）。设置按钮不携带任何状态信息，绿色边框只是视觉噪音。
**建议**：统一为与刷新按钮相同的 `border-slate-200 text-slate-700`，两个按钮保持一致的中性风格。

### 31.【P1】设置选项排序不符合使用频率
**截图实证**：设置首屏显示的 3 项为：语言、开机自启、刷新间隔。
**问题**：语言和开机自启是一次性设置项，设定后几乎不会再改。而"托盘摘要规则"和"菜单栏服务"是用户更常调整的显示偏好，却被排在第二组需要滚动才能看到。
**建议**：按使用频率重排首屏内容，将高频操作（摘要规则、菜单栏服务）提前，低频操作（语言、自启）后移。

### 32.【P1】设置页卡片宽度与面板卡片不一致
**代码实证**：面板 ServiceCard 直接放在容器 `grid gap-4` 中，占满可用宽度。设置页 PreferenceField 包裹在 PreferenceSection 中，PreferenceSection 有 `px-1`（`PreferenceSection.tsx:9`），导致设置卡片比面板卡片窄约 8px。
**问题**：两个视图中的卡片宽度不一致，来回切换时视觉上不统一。
**建议**：去掉 PreferenceSection 的 `px-1`，或统一两侧容器的水平 padding。

### 33.【P1】设置-状态区域缺少 Claude Code 连接状态
**代码实证**：`SettingsView.tsx:304-316`，状态区域仅展示 `panelState?.activeSession`（Codex 服务的会话信息），`claudeCodePanelState` 在 `useAppState()` 中可用但未在此区域使用。
**问题**：用户可能只使用 Claude Code 而不使用 Codex，此时状态区域会显示"无活动会话"，造成困惑。状态区域应展示所有已配置服务的连接状态。
**建议**：遍历所有服务，展示各自的连接状态、数据来源和会话信息。

### 34.【P1】"发送测试通知"按钮应移除
**代码实证**：`SettingsView.tsx:325-335`，该按钮调用 `sendTestNotification()` 发送一条测试通知。
**问题**：这是一个调试功能，正式产品中用户没有测试通知的需求。如果通知不工作，用户也不会通过这个按钮来排查。该按钮占据设置页空间且增加认知负担。
**建议**：直接移除。如需保留调试能力，可通过开发者快捷键或命令行参数触发。
**与 pre-009 #17 的关系**：#17 建议降低视觉权重（已实施），本条进一步建议完全移除。

### 35.【P1】拖拽排序在 WKWebView 中可能不工作
**代码实证**：`SettingsView.tsx:207-216`，拖拽使用 HTML5 Drag & Drop API（`draggable`、`onDragStart`、`onDrop`）。
**问题**：macOS WKWebView 对 HTML5 Drag & Drop 的支持不如 Chromium 完善。实际操作中拖拽手柄显示 hover 阴影效果，但拖拽动作可能无响应。原因可能包括：WKWebView 拖拽事件兼容性、Tauri 窗口拦截拖拽事件、或缺少 `dragenter` 处理。
**建议**：改用 Pointer Events 实现拖拽（`onPointerDown`/`onPointerMove`/`onPointerUp`），不依赖 HTML5 Drag & Drop API。这在所有 WebView 引擎中都能可靠工作。

### 36.【P2】英文模式下健康摘要可能被截断
**代码实证**：`AppShell.tsx:251`，摘要文本使用 `truncate`。面板 header 左侧可用宽度约 256px（360px - 24px padding - 80px 按钮区），英文摘要如 "Claude Code weekly quota running low" 可能超出并被 ellipsis 截断。
**问题**：截断后用户无法获取完整信息，削弱了健康摘要的价值。
**建议**：缩短英文摘要文案（如 "Claude Code weekly low"），或调整 header 布局为摘要文本留出更多空间。

### 38.【P0】进度条内文字重复
**截图实证**：进度条内左侧显示 `47%`，右侧显示 `剩余 47%`（英文 `47% remaining`），同一信息重复出现。
**问题**：009 迭代实施 pre-009 #1/#3（百分比嵌入进度条）时，左右各放了一份百分比，产生冗余。用户需要扫视两段文字才能确认它们表达的是同一件事，违背"一眼扫视"的设计目标。
**建议**：仅保留一条文字（如 `剩余 47%`），居中或靠左显示。需确保文字在进度条填充区和空白区两种背景上都清晰可读（对比度）。

### 39.【P0】手动/自动刷新后菜单栏数据不更新
**问题**：面板刷新（手动点击或定时自动）调用的 `refresh_codex_panel_state` 和 `refresh_claude_code_panel_state` 没有调用 `apply_display_mode`，导致菜单栏仍显示旧数据。只有偏好设置变更（`save_preferences`、`set_autostart`）才会更新菜单栏。
**影响**：用户在面板中看到最新额度，但菜单栏显示的仍是上次偏好变更时的旧数据，两处信息不一致。
**建议**：在两个刷新命令成功后调用 `apply_display_mode` 同步菜单栏。需为命令增加 `AppHandle` 参数。
**与 #37 的关系**：#37 是菜单栏展示层增强（图标、颜色），本条是数据同步缺陷，更基础，应优先修复。

### 37.【P1】菜单栏显示缺少服务区分和颜色警告
**截图实证**：菜单栏当前显示 `[app icon] 37% / 32%`，无法区分这是哪个服务的额度，也无法从颜色判断是否需要关注。
**代码实证**：`tray/mod.rs:182-189`，菜单栏仅通过 `set_title(text)` 设置纯文本，`set_tooltip` 设置提示。没有根据额度状态切换图标或颜色。
**问题**：
1. 用户无法从菜单栏图标判断当前展示的是哪个 AI 服务的额度
2. 37%/32% 均低于 50%，但菜单栏文字无任何视觉警告
**建议**：
- 根据当前 `menubarService` 设置显示对应的服务图标（Claude / Codex），或在 tooltip 中标明服务名
- 根据最低额度状态切换菜单栏图标颜色（正常/warning/danger），可通过准备多套 tray icon 资源实现
