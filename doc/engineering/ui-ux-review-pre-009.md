## 一、面板视图（Panel View）

### 1.【P0】进度条太细，不够"一眼扫视"
**截图实证**：进度条高度 `h-3`（12px），百分比数字（94%、39% 等）在上方用 `text-xs` 小字展示，与维度标签（5h、week）同行。
**问题**：PRD 核心价值是"打开弹窗 < 1 秒即可看到所有服务余量"。当前进度条在视觉层级中权重不足——用户需要先读数字，再看进度条确认，而非一眼扫到颜色+粗细就获得直觉判断。
**建议**：
- 加粗到 `h-5`（20px），在条内嵌入百分比文字（类似 macOS 存储管理条）
- 或改为环形图/半圆仪表盘，对"余量"概念更直觉
- 考虑在进度条旁同时显示绝对值（如 "42% · 剩余 128 次请求"），PRD 4.2 明确要求"百分比 + 绝对值"

### 2.【P0】面板头部"额度面板"是无效信息
**截图实证**：Header 左侧显示"额度面板"四个字，占据 sticky 头部最醒目的位置。
**问题**：用户已经知道自己打开的是额度面板，这四个字零信息量。PRD 3 的核心指标是"< 1 秒看到余量"，头部应该为此服务。
**建议**：替换为**整体健康度摘要**：
- 全部充裕时：`所有服务正常 ✓`
- 有维度偏低时：`⚠ Claude Code week 额度偏低`
- 用户甚至不用滚动到具体卡片就知道是否需要关注

### 3.【P0 ↑】百分比数字扫视效率低
**截图实证**：94%、39%、100%、63% 四个数字分布在不同卡片的不同行中，字号 `text-xs`，与标签文字同色同大小，视觉上被淹没。
**问题**：这是用户最关心的核心数据，但在当前设计中没有被视觉强调。用户需要逐行扫读才能找到关键数字。
**建议**：
- 百分比使用 `text-sm font-bold` 或更大字号，与标签拉开层级
- 低余量（< 50%）的百分比使用对应的警告色（amber/red），而非统一的 slate 色
- 如果采用进度条内嵌数字方案，此项可合并到 #1

### 4.【P1】缺少服务品牌图标
**截图实证**：卡片头部只有纯文字"Claude Code"、"Codex"，旁边是绿色"实时"徽章。
**PRD 对照**：4.2 明确要求"每个已配置服务的**名称、图标**"。
**建议**：在服务名左侧加 18–20px 品牌图标（Claude 圆标 / Codex logo）。图标能帮助用户在多卡片中快速定位目标服务。

### 5.【P1】"实时"徽章含义不明确
**截图实证**：两张卡片都显示绿色"实时"徽章。
**问题**：从用户视角，"实时"不是一个清晰的状态概念——是指数据是实时的？还是服务是实时可用的？当它在所有卡片上都出现且都是绿色时，用户会忽略它，变成视觉噪音。
**建议**：
- 明确徽章语义：如果是数据源类型，改为"OAuth"/"本地"；如果是连接状态，用图标（绿点）替代文字
- 仅在异常时才显示（如"离线"、"过期"），正常状态不占空间

### 6.【P1】时间显示格式不一致
**截图实证**：同一张卡片内，重置提示用相对时间（"4 小时后重置"、"4 天后重置"），但"上次刷新"用绝对时间（"2026/3/23 10:19:29"）。
**问题**：同一视觉区域内两种时间格式混用，增加认知负担。相对时间是正确的设计方向（PRD 强调"< 1 秒"扫视），但没有贯彻到底。
**建议**：
- "上次刷新"改为相对时间（"3 分钟前"、"刚刚"），hover tooltip 显示绝对时间
- 与重置提示保持同一格式体系

### 7.【P1】"上次刷新"每卡重复，占空间
**截图实证**：Claude Code 和 Codex 卡片底部各有一行"上次刷新: 2026/3/23 10:19:29"，时间相同。
**问题**：当刷新是全局操作时，每卡重复显示是信息冗余。未来 4-5 个服务时，冗余更明显。
**建议**：
- 移到面板底部作为全局信息："上次刷新：3 分钟前"
- 仅当各服务刷新时间不同时，才在卡片级显示

### 8.【P1】额度维度标签不够友好
**截图实证**：显示 "5h"、"week" 缩写。
**问题**：对于不熟悉 AI 服务计费模型的用户，"5h" 的含义不直观（是 5 小时窗口？最近 5 小时？）。
**建议**：映射为"5 小时窗口"、"每周额度"等完整标签。空间不足时可用"5h 窗口"、"周额度"折中。

### 9.【P1】卡片无紧急状态视觉提升
**截图实证**：Claude Code 的 week 维度仅 39%（amber），但整张卡片的视觉权重与 100% 的 Codex 卡片完全相同。
**问题**：用户打开面板后，需要逐一阅读每个维度才能发现问题。PRD 目标是"< 1 秒看到余量"，有紧急状态时应该更快被注意到。
**建议**：
- 卡片级紧急状态：当某维度 < 50% 时，卡片左侧加 3px 彩色边条（amber/red），或卡片背景微调
- 与 #2（头部健康度摘要）联动：头部告警 → 对应卡片高亮

### 10.【P1】未连接服务的占位卡缺少行动引导
**截图实证**：当前截图两个服务均已连接，未展示空状态。从代码确认：未连接时显示虚线框 + 纯文字描述，无操作入口。
**建议**：
- 加入 CTA 按钮或链接，如"前往设置 →"或"查看连接指引"
- 区分"未安装"和"已安装但未登录"，给出不同引导
- 与 #20（首次使用 Onboarding）配合

### 11.【P2】颜色是区分余量状态的唯一手段
**截图实证**：绿色（94%、100%）、amber（39%、63%）进度条。红色在截图中未出现但低于 20% 时触发。
**问题**：色盲用户无法区分绿/黄/红。
**建议**：低余量时在进度条旁加文字标签（"偏低"、"紧张"）或警告图标 ⚠。

## 二、设置视图（Settings View）

### 12.【P0】保存按钮模式过时 → 改为即时生效
**截图实证**：截图仅展示设置页上半部分（语言、刷新间隔、托盘摘要、自启动、菜单栏服务），保存按钮在下方不可见。从代码确认：修改后需手动点击"保存偏好"，但 autostart toggle 已是即时生效。
**问题**：两种交互模式混用（toggle 即时生效 vs 其他需手动保存），用户会困惑"我改了语言，生效了吗？"。
**建议**：
- 所有非危险设置 → 修改即保存（语言、刷新间隔、摘要模式、菜单栏服务）
- 仅网络代理（需验证 URL 格式）保留显式"应用"按钮
- 删除通用"保存偏好"按钮，统一为 auto-save

### 13.【P0】服务排序 → 直接支持拖拽排序
**截图实证**：设置页中的服务排序在截图可见范围之外。从代码确认：使用 ↑ ↓ 文字按钮。
**问题**：文字按钮的排序方式视觉原始，与整体 UI 精致感不匹配。
**建议**：实现拖拽排序。左侧加拖拽手柄（⠿ grip icon），支持鼠标拖拽。无论当前两个服务还是未来更多服务，拖拽都是最自然的排序方式。

### 14.【P1】设置页过长，超出视窗
**截图实证**：620px 高的窗口只能显示前 5 个设置项（语言、刷新间隔、托盘摘要、自启动、菜单栏服务），服务排序、代理、保存按钮、通知测试、Codex CLI 信息全部在折叠区需要滚动。
**问题**：用户打开设置页后看不到保存按钮，可能不知道更改需要保存（如果保留保存模式的话）。更重要的是，关键设置被推到不可见区域。
**建议**：
- 将设置分组：「通用」（语言、自启动）|「显示」（摘要模式、服务排序、菜单栏服务）|「连接」（代理、刷新间隔）
- 分组可折叠，或用 tab 切换，控制单屏信息量
- "Codex CLI" 区域是连接**状态**不是设置，应视觉分离或移到面板

### 15.【P1】"返回"按钮缺少方向图标
**截图实证**：设置页右上角有文字按钮"返回"，无箭头图标。
**问题**：macOS 原生 popover 和 iOS 均使用 `← 返回` 的模式。纯文字"返回"缺少方向暗示，尤其在面板这种从右向左的导航模式中。
**建议**：在"返回"文字左侧加 `‹` 或 `←` 箭头图标，或改为图标按钮。

### 16.【P1】底部 Codex CLI 信息卡视觉风格突兀
**截图实证**：不在当前截图可见范围，从代码确认：底部出现深色背景（暗绿/墨绿色）信息卡片。
**问题**：与整体白色/浅色卡片风格不统一，视觉跳跃感强。
**建议**：改为浅色 info banner（白底 + 浅灰背景 + 左侧彩色边条），与整体设计语言一致。

### 17.【P2】"通知测试"按钮放置和视觉权重不当
**截图实证**：不在当前截图可见范围。从代码确认：amber 色"发送测试通知"与深绿色"保存偏好"视觉权重接近。
**问题**：通知测试是低频调试功能，不应与主操作按钮平级。
**建议**：移到设置页最底部，改为 ghost/outline 按钮降低视觉权重。

## 三、视觉与微交互

### 18.【P1】刷新按钮缺少旋转动画
**截图实证**：面板头部右侧有两个小图标按钮（刷新 + 设置齿轮），刷新中时按钮仅变灰（`disabled:opacity-60`）。
**问题**：用户点击刷新后没有"正在工作"的视觉反馈，不确定操作是否生效。
**建议**：刷新中给 RefreshIcon 加 `animate-spin` 旋转动画。刷新完成后可短暂显示 ✓ 或恢复静止。

### 19.【P1】Header 滚动时缺少分隔指示
**截图实证**：sticky header（"额度面板" + 按钮）与下方内容之间无明显分隔。当内容滚动到 header 下方时，视觉上粘连。
**建议**：滚动时给 header 底部加 1px 分隔线或轻微 `shadow-sm`，未滚动时无。

### 20.【P2】首次使用无 Onboarding
**现状**：新用户首次打开看到空状态（虚线占位卡），不知下一步该做什么。
**建议**：首次启动显示 2-3 步简洁引导（"连接你的第一个 AI 服务"），引导到设置页。

### 21.【P2】视图切换无过渡动画
**截图实证**：面板 → 设置 → 面板，均为硬切（无动画）。
**建议**：加 slide 或 fade 过渡。设置从右滑入，返回时从右滑出，符合 iOS/macOS 导航心智模型。

### 22.【P2】进度条宽度变化缺少明确动画时长
**现状**：CSS 有 `transition-[width]` 但无 duration，浏览器使用默认值。
**建议**：加 `duration-500 ease-out`，数据刷新时进度条平滑过渡。

### 23.【P2】CSS 伪毛玻璃效果无实际作用
**截图实证**：面板背景为 `bg-white/90 backdrop-blur-sm`，但 Tauri 窗口未配置 transparent/vibrancy，所以 blur 不模糊桌面。
**建议**：二选一：
- **方向 A（推荐）**：去掉伪 blur，改为纯色背景（`bg-white` 或微妙渐变），减少 GPU 开销
- **方向 B**：启用 Tauri `transparent: true` + macOS `vibrancy`，实现真正系统级毛玻璃

### 24.【P2】Header 图标按钮触控目标偏小
**截图实证**：头部右侧的刷新和设置按钮为 `h-8 w-8`（32×32px），图标本身更小。
**问题**：作为面板级核心操作（刷新、进入设置），触控目标偏小。尤其在菜单栏 popover 这种快速操作场景中，用户可能用较快的手势点击。
**建议**：增大到 `h-9 w-9` 或 `h-10 w-10`（36–40px），或加大 padding 扩展可点击区域。

## 五、与首轮审查的变化说明

本轮基于真实 Tauri 窗口截图，对首轮（2026-03-22，基于 Playwright + demo 数据）做了以下调整：

| 变化 | 说明 |
|------|------|
| **方法升级** | 从 Playwright demo 页截图 → macOS 原生 API 真实窗口截图 |
| **移除 demo 局限** | 首轮"已知局限"中的 4 项 demo 问题不再存在 |
| **#3 新增 P0** | 百分比数字扫视效率低——在真实数据（94%/39%/100%/63%）下更明显 |
| **#5 新增** | "实时"徽章含义不明确——demo 模式中未注意到此问题 |
| **#6 新增** | 时间格式不一致——重置提示用相对时间 vs 上次刷新用绝对时间 |
| **#7 新增** | "上次刷新"每卡重复——真实数据下两卡时间相同，冗余感明显 |
| **#9 新增** | 卡片无紧急状态提升——39% 的 amber 状态在真实场景中缺乏视觉突出 |
| **#15 新增** | "返回"按钮缺少方向图标 |
| **#24 新增** | Header 图标按钮触控目标偏小 |
| **原 #4 → #6** | 合并"上次刷新时间格式"到更广泛的"时间显示不一致"问题 |
| **原 #5 → #8** | 维度标签建议不变 |
