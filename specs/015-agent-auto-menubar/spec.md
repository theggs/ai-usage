# Feature Specification: 菜单栏自动服务跟随

**Feature Branch**: `015-agent-auto-menubar`  
**Created**: 2026-03-27  
**Status**: Draft  
**Input**: User description: "根据 doc/engineering/015-agent-activity-passive-detection-research.md 的背景调查，定义下一次迭代规格。增加需求：1. 在设置-菜单栏服务中增加 `自动` 选项。选择自动后，在菜单栏轮播显示最近活动的agent用量。2. 当菜单栏中显示余量数据时，把菜单栏显示的图标修改为使用当前ai服务的图标。"

## Clarifications

### Session 2026-03-27

- Q: 自动模式到底应该是哪一种语义？ → A: 混合模式：默认只显示一个当前服务，但如果两个服务都在近期活跃，则在它们之间低频轮播
- Q: 自动模式下“无近期活动”时显示什么？ → A: 保留上一次显示的服务数字和图标，直到新的近期活动出现
- Q: “近期活跃”如何定义？ → A: 最近 5 分钟内存在可信活动信号；低频轮播指轮播间隔不短于 15 秒

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 自动跟随最近活跃服务 (Priority: P1)

作为同时使用 Codex 和 Claude Code 的用户，我希望在“菜单栏服务”里选择“自动”后，菜单栏能够优先显示最近 5 分钟内更活跃的 AI 服务；当两个服务都在最近 5 分钟内处于近期活跃时，再以低频轮播的方式显示它们的余量，而不是要求我频繁手动切换服务。

**Why this priority**: 这是本次迭代的核心价值。如果自动模式不能替代手动切换，设置里新增“自动”选项就没有实际意义。

**Independent Test**: 启用“自动”后，分别验证“只有一个服务在最近 5 分钟内活跃”和“两个服务都在最近 5 分钟内活跃”两类场景；无需进入设置页修改选项，菜单栏应在前一种场景下稳定显示单一服务，在后一种场景下以不短于 15 秒一次的节奏低频轮播显示两个服务。

**Acceptance Scenarios**:

1. **Given** 用户打开设置页并查看“菜单栏服务”，**When** 用户展开该选项，**Then** 可见 `Codex`、`Claude Code` 和 `自动` 三种模式。
2. **Given** 用户已把“菜单栏服务”设置为 `自动`，**When** 最近活跃服务从 Codex 变为 Claude Code，**Then** 菜单栏余量显示对象会从 Codex 切换为 Claude Code，而无需用户再次进入设置。
3. **Given** 用户已把“菜单栏服务”设置为 `自动`，**When** 只有一个服务在最近 5 分钟内存在明确的活动信号，**Then** 菜单栏稳定显示该服务，而不会在两个服务之间来回切换。
4. **Given** 用户已把“菜单栏服务”设置为 `自动`，**When** 两个服务都在最近 5 分钟内存在活动信号，**Then** 菜单栏会以不短于 15 秒一次的稳定节奏在这两个服务之间轮播显示，而不是高频闪烁或随机跳变。
5. **Given** 用户已把“菜单栏服务”设置为 `自动`，**When** 两个服务都超过 5 分钟没有新的活动信号，**Then** 菜单栏继续保留上一次已显示的服务数字与图标，不出现频繁闪烁、空转或随机切换。

---

### User Story 2 - 通过图标一眼识别当前菜单栏显示对象 (Priority: P1)

作为需要快速扫视菜单栏的用户，我希望当菜单栏显示某个服务的余量时，图标也同步切换成该服务的图标，这样我不需要打开面板就能知道当前数字属于哪个服务。

**Why this priority**: 自动切换余量后，如果图标仍然保持同一个通用样式，用户很容易误读当前数字的归属服务。图标同步是自动模式可理解性的关键组成部分。

**Independent Test**: 分别让菜单栏进入显示 Codex 余量和显示 Claude Code 余量两种状态，直接观察菜单栏图标与数字组合，用户应能立刻识别当前归属服务。

**Acceptance Scenarios**:

1. **Given** 菜单栏当前显示的是 Codex 余量，**When** 用户查看菜单栏图标与数字，**Then** 图标使用 Codex 服务图标。
2. **Given** 菜单栏当前显示的是 Claude Code 余量，**When** 用户查看菜单栏图标与数字，**Then** 图标使用 Claude Code 服务图标。
3. **Given** 菜单栏当前尚未形成任何可显示的服务对象，**When** 用户查看菜单栏图标，**Then** 图标保持统一的中性应用图标，而不是误显示某个服务图标。
4. **Given** 自动模式导致当前显示对象发生变化，**When** 菜单栏余量数字更新为新的服务，**Then** 图标与数字归属服务同步变化，不出现“数字已经切到新服务、图标还停留在旧服务”的状态。

---

### User Story 3 - 自动模式仍然保持可控和可预期 (Priority: P2)

作为希望兼顾自动化和可控性的用户，我希望自动模式只在最近 5 分钟内的活动信号足够明确时切换或轮播菜单栏显示对象；当自动判断不稳定、服务不可用或我更偏好固定显示某个服务时，仍然可以回到手动模式。

**Why this priority**: 自动跟随只有在结果稳定、边界清晰时才会被用户信任。若自动模式经常误切换或破坏已有手动控制，用户会直接关闭该能力。

**Independent Test**: 在自动模式下验证服务断开、长期无活动、单服务可用、手动切回固定服务等场景；菜单栏行为应稳定且符合用户预期。

**Acceptance Scenarios**:

1. **Given** 用户当前选择的是 `自动`，**When** 某个服务没有可用余量数据或当前不可读，**Then** 自动模式不会把该服务纳入菜单栏轮播并展示误导性数字。
2. **Given** 用户当前选择的是 `自动`，**When** 只有一个服务在最近 5 分钟内存在可判断的近期活动且可显示余量，**Then** 菜单栏只显示该服务的余量与对应图标，而不进入轮播。
3. **Given** 用户当前选择的是 `自动`，**When** 两个服务都在最近 5 分钟内存在可判断的近期活动且都可显示余量，**Then** 菜单栏只在这两个服务之间以不短于 15 秒一次的节奏轮播，而不会把不满足条件的服务纳入轮播。
4. **Given** 用户当前选择的是 `自动`，**When** 用户改回固定的 `Codex` 或 `Claude Code`，**Then** 菜单栏立即恢复到手动模式，并停止受自动跟随结果影响。
5. **Given** 用户重新启动应用，**When** 打开设置页，**Then** 上一次保存的“菜单栏服务”模式仍然被保留，包括 `自动` 模式。
6. **Given** 用户当前选择的是 `自动` 且菜单栏之前已经显示过某个服务，**When** 两个服务随后都长时间没有新的近期活动，**Then** 菜单栏继续保留上一次已显示的服务数字与图标，直到新的近期活动重新改变当前显示对象。

### Edge Cases

- 自动模式下，两种服务都在最近 5 分钟内产生可信活动时，菜单栏允许以不短于 15 秒一次的节奏低频轮播，但不应高频闪烁或频繁无序跳变。
- 自动模式下，当前显示服务变更时，图标与数字必须同步切换，不得出现短暂错位。
- 自动模式下，如果最近活跃服务当前没有可展示的余量数据，不应显示过期或错误归属的数字。
- 自动模式下，如果某一时刻没有任何可判断的近期活动且之前已经形成过显示结果，菜单栏应继续保留上一次显示的服务，而不是在中性态与某个服务之间反复跳动。
- 自动模式下，如果用户刚进入自动模式且系统尚未形成任何可显示的服务对象，菜单栏应保持中性图标，不得伪造某个服务为当前显示对象。
- 自动模式下，如果近期活跃服务从“双服务都活跃”收敛为“只剩一个服务在最近 5 分钟内活跃”，菜单栏应退出轮播并稳定显示该单一服务。
- 手动模式下，图标应继续与当前手动选择的服务保持一致；自动跟随结果不得影响手动模式。
- 当菜单栏当前不展示具体服务数字时，不得误用某个服务图标暗示其为当前显示对象。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须在设置页的“菜单栏服务”选项中提供 `自动` 模式，与现有固定服务模式并列。
- **FR-002**: 用户必须能够像选择固定服务一样选择 `自动` 模式，并且该选择必须被持久化保存。
- **FR-003**: 当“菜单栏服务”为 `自动` 时，系统必须依据最近 5 分钟内的服务活动情况决定菜单栏当前显示对象。
- **FR-004**: 自动模式的判断依据必须来自本机可读取的近期活动痕迹，不得要求用户额外授予系统权限、安装 hook 或改变日常使用命令。
- **FR-005**: 自动模式必须优先在已支持的服务范围内工作；本次迭代支持范围为 Codex 与 Claude Code。
- **FR-006**: 自动模式下，当只有一个服务在最近 5 分钟内存在足够明确的活动信号时，系统必须稳定显示该服务，而不是进入轮播。
- **FR-007**: 自动模式下，当多个已支持服务都在最近 5 分钟内存在足够明确的活动信号且都可显示余量时，系统必须允许在这些服务之间以不短于 15 秒一次的低频轮播方式展示菜单栏余量。
- **FR-007a**: 自动模式的轮播范围必须只包含同时满足“近期活跃”与“可显示余量”条件的服务，不得把不满足条件的服务纳入轮播。
- **FR-007b**: 自动模式必须避免因短时间内重复或交错的活动痕迹导致菜单栏高频闪烁、随机跳变或无意义轮换。
- **FR-008**: 当自动模式判断出的当前服务没有可展示的余量数据时，系统不得显示错误归属、过期或误导性的数字。
- **FR-009**: 当用户把“菜单栏服务”从 `自动` 切回固定服务时，系统必须立即恢复到手动模式，并停止根据最近活动自动切换。
- **FR-010**: 当菜单栏当前显示某个服务的余量数字时，系统必须同步使用该服务的图标作为菜单栏图标。
- **FR-011**: 当菜单栏当前显示对象从一个服务切换到另一个服务时，系统必须保证图标与数字的归属服务同步变化。
- **FR-012**: 当自动模式下没有新的近期活动且系统已经存在上一次显示结果时，系统必须继续保留上一次显示的服务数字与图标，直到新的近期活动重新决定当前显示对象。
- **FR-012a**: 当菜单栏当前尚未形成任何可显示的服务对象时，系统必须使用中性应用图标，而不是默认落到某个服务图标。
- **FR-013**: 手动模式下，菜单栏图标也必须与当前手动选择的服务保持一致，确保图标与数字表达同一个服务对象。
- **FR-014**: 自动模式不得改变主面板中各服务卡片的展示顺序或内容；其作用范围仅限于菜单栏当前显示对象与轮播集合的决定。
- **FR-015**: 菜单栏服务模式必须在应用重启后保持可恢复；如果用户上次保存的是 `自动`，应用重启后仍应以 `自动` 模式工作，并重新根据本机活动信号决定菜单栏当前显示对象。

### Key Entities *(include if feature involves data)*

- **菜单栏服务模式**: 用户对菜单栏显示逻辑的选择，包含固定显示某个服务与自动跟随最近活跃服务两类模式。
- **服务活动状态**: 某个已支持服务在当前设备上的最近活动判断结果，至少包含最近活动时间、当前是否可用于菜单栏自动切换以及判断置信度。
- **菜单栏当前显示对象**: 菜单栏当前用于显示数字与图标的服务对象，可由手动模式直接指定，也可由自动模式动态决定。
- **菜单栏服务图标状态**: 菜单栏当前使用的图标语义；在存在当前显示对象时必须与其一致，在尚未形成任何显示对象时回退为中性图标。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户进入设置页后，能够在 10 秒内发现并理解“菜单栏服务”中的 `自动` 选项，无需阅读额外说明文档。
- **SC-002**: 在自动模式下，用户从主要使用一个服务切换为主要使用另一个服务后，菜单栏会在 2 分钟内更新为新的服务余量与对应图标；若两个服务都在最近 5 分钟内处于近期活跃状态，则会进入不短于 15 秒一次的低频轮播。
- **SC-003**: 在自动模式下的连续使用场景中，菜单栏不会因短时间内交错的活动痕迹产生持续性闪烁或肉眼可见的高频来回切换；允许存在稳定、低频、可理解的轮播。
- **SC-004**: 在 Codex、Claude Code 和中性态 3 类菜单栏状态下，E2E 与截图复核都必须确认图标与数字归属一致，不存在“数字属于一个服务、图标属于另一个服务”的错配。
- **SC-005**: 用户把菜单栏服务从 `自动` 切换回固定服务后，菜单栏会在当前设置生效周期内恢复稳定的手动显示，不再受到自动判断影响。
- **SC-006**: 在自动模式下，如果两个服务都停止产生近期活动，菜单栏会继续稳定保留上一次显示结果，不会在空状态、中性图标和旧服务之间来回抖动。

## Assumptions

- 本次迭代的“自动”语义定义为混合模式：默认只显示一个当前服务；当多个服务都在最近 5 分钟内处于近期活跃且都可显示余量时，在这些服务之间以不短于 15 秒一次的节奏低频轮播。
- 当自动模式暂时没有新的近期活动时，如果菜单栏之前已经形成过显示结果，则继续保留上一次显示的服务数字与图标；只有在尚未形成任何显示对象时才使用中性图标。
- 自动模式只影响菜单栏的当前显示对象，不改变主面板中的卡片顺序、服务详情内容或刷新入口位置。
- 本次迭代不新增新的 AI 服务类型，自动模式只在当前已支持的 Codex 与 Claude Code 之间工作。
- 自动模式的判断来源限定为本机可读取的近期活动痕迹；“近期活跃”统一定义为最近 5 分钟内存在可信活动信号，不要求用户额外授权系统权限，也不要求用户改动启动命令或安装额外集成。
