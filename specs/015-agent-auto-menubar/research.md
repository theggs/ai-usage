# 研究：菜单栏自动服务跟随

**特性**：015-agent-auto-menubar  
**日期**：2026-03-27  
**规格**：[spec.md](./spec.md)

## 背景

本特性的产品目标不是识别前台窗口，而是在尽量不打扰用户的前提下，让菜单栏优先显示“最近正在使用的 AI 服务”的余量。结合 PRD 的 local-first 方向和既有研究结论，本次计划选择继续沿用“纯被动读取本机活动痕迹”的路线，不引入额外权限、shell hook 或主动上报。

---

## 决策 1：自动模式由 Rust 宿主层负责活动判断与菜单栏切换

- **Decision**：在 `src-tauri/` 新增宿主侧活动监测模块，由 Rust 负责读取本地活动痕迹、计算当前菜单栏显示对象，并直接驱动 tray 图标与标题更新。
- **Rationale**：
  - tray 生命周期归宿主所有，窗口隐藏时也要继续工作，不能依赖前端页面是否处于可见态。
  - 本地文件读取属于受信边界，符合宪章的 host-boundary security 要求。
  - 把“活动判断”放到前端会造成重复状态源，增加 tray 与 panel 不一致的风险。
- **Alternatives considered**：
  - 前端定时轮询后再回调宿主更新 tray：窗口隐藏或初始化时不稳，且会把文件系统读取扩散到前端编排层。
  - 继续只在 `save_preferences` / 手动刷新时刷新 tray：无法满足自动模式的持续切换需求。

## 决策 2：Codex 使用只读 SQLite + JSONL/mtime 的组合信号

- **Decision**：Codex 主信号使用 `~/.codex/state_5.sqlite` 的 `threads.updated_at`；辅助信号使用 `session_index.jsonl.updated_at`、`logs_1.sqlite` 的最新 `ts` 与 `sessions/**/*.jsonl` 的最新 mtime。宿主侧增加 `rusqlite` 作为只读查询依赖。
- **Rationale**：
  - `threads.updated_at` 是当前可见范围里最结构化、最接近会话活跃度的字段。
  - JSONL 与文件 mtime 可在 SQLite 文件锁定或结构异常时作为降级链路，避免“读不到一次就彻底失能”。
  - 使用 `rusqlite` 比依赖外部 `sqlite3` 命令更可测试，也避免引入新的 shell 执行边界。
- **Alternatives considered**：
  - 只看 `sessions/**/*.jsonl` mtime：成本低，但对多会话和后台整理动作的区分太弱。
  - 只看 `session_index.jsonl`：足够轻量，但在需要判断“仍在持续活跃”时不如 SQLite 稳定。

## 决策 3：Claude Code 使用项目会话 JSONL + history 的组合信号

- **Decision**：Claude Code 主信号使用 `~/.claude/projects/**/*.jsonl` 的最新文件 mtime；辅助信号使用 `history.jsonl` 最新 `timestamp`，`session-env/<sessionId>` 目录 mtime 仅作为弱回退。
- **Rationale**：
  - `projects/**/*.jsonl` 最接近实际项目会话，且更新时间与用户交互更相关。
  - `history.jsonl` 可以提供全局时间线补充，适合在项目会话路径不可读时做降级。
  - `session-env` 目前没有稳定结构，不适合作为主信号，但保留为弱辅助有助于兜底。
- **Alternatives considered**：
  - 只读 `history.jsonl`：覆盖面广，但对当前项目会话的针对性不够。
  - 解析正文内容或 prompt：超出产品目标，也违反“只读取必要元数据”的边界。

## 决策 4：自动模式使用“单服务优先 + 双服务低频轮播 + 无活动保留上次结果”的选择模型

- **Decision**：宿主维护一个运行时自动选择状态：
  - 只有一个服务存在明确近期活动时，稳定显示该服务。
  - 两个服务都处于近期活跃且都能显示余量时，进入低频轮播。
  - 两个服务都没有新的近期活动时，保留上一次显示的服务数字与图标。
  - 若用户刚进入自动模式且尚未形成过任何显示结果，则使用中性图标。
- **Rationale**：
  - 这与已澄清的规格保持一致，也最符合“减少手动切换但不制造随机闪动”的体验目标。
  - 保留上次结果比回到空态更符合“继续工作流”的产品起点，同时仍保持 truthful，因为它代表的是“最近一次自动选择结果”，不是伪造新活动。
- **Alternatives considered**：
  - 始终固定轮播所有服务：会降低可读性，也不符合“当前主要在用哪个服务”的目标。
  - 无活动立即回到中性态：更保守，但会造成菜单栏频繁退回空态，增加理解成本。

## 决策 5：活动扫描与轮播使用轻量定时器，不触发额外远端额度刷新

- **Decision**：宿主增加一个轻量后台循环，只做本地活动信号读取和 tray 选择更新。建议参数：
  - 活动扫描周期：15 秒
  - 活跃窗口：120 秒
  - 明确切换阈值：另一服务至少领先 15-30 秒，或拥有更高置信度主信号
  - 轮播间隔：15 秒
- **Rationale**：
  - 规格要求 2 分钟内完成跟随，15 秒扫描足以满足，同时不会把本地 IO 频率推高到影响空闲 CPU。
  - 活动判断与远端额度刷新是两条不同链路；自动模式不能为了换显示对象而额外请求 Codex/Claude 官方数据。
  - 轮播节奏需要独立于刷新间隔，否则用户只会在 5-15 分钟的大刷新后才看到切换。
- **Alternatives considered**：
  - 复用现有 `refreshIntervalMinutes`：过慢，不满足自动模式的交互预期。
  - 文件系统 watch：跨平台和跨目录稳定性不如轮询，也更难处理 SQLite 与 JSONL 混合来源。

## 决策 6：tray 图标改为“服务图标基底 + 严重度着色”，中性态回退应用图标

- **Decision**：菜单栏/托盘图标改为跟随当前显示服务的图标基底；在存在具体服务显示对象时，使用该服务的 tray 资产并保留现有严重度着色能力；在尚未形成任何显示对象时，回退到中性应用图标。
- **Rationale**：
  - 规格明确要求图标与数字同步表达同一个服务。
  - 现有 tray 已依赖图标颜色表达风险等级，完全丢弃严重度会削弱原有提示能力；因此更合适的是“换服务基底，不放弃风险着色”。
  - 采用预置的宿主侧 raster 资产比运行时解析前端 SVG 更稳，也避免新增图像解析依赖。
- **Alternatives considered**：
  - 继续使用统一应用图标，只改数字：会导致归属关系不清。
  - 直接在宿主解析前端 SVG：实现成本更高，跨平台渲染一致性更差。

## 决策 7：自动模式只影响菜单栏，不改变面板卡片顺序与内容

- **Decision**：自动模式的作用域只限于 tray 当前显示对象、图标和 tooltip；主面板中的服务卡片顺序、详情内容、刷新入口与已有状态表达保持不变。
- **Rationale**：
  - 这是规格和 PRD 共同约束的边界，能显著降低本轮改动面和回归风险。
  - 面板是“看全量状态”的地方，菜单栏是“看当前最相关状态”的地方，两者不需要共享同一个呈现排序。
- **Alternatives considered**：
  - 让面板卡片也跟着自动模式重排：会改变既有信息架构，超出本轮需求。

## 决策 8：故障与缺失信号按“降级而非伪造”处理

- **Decision**：任一信号源不可读、结构异常或文件暂时锁定时，宿主只把该信号标记为不可用，并继续尝试该服务的回退链路；若两个服务都拿不到可用高/中置信度信号，则维持上一次稳定结果或中性态，不伪造新活动。
- **Rationale**：
  - 本特性必须满足 truthful state，不能因为读取失败就伪造“当前正在用某服务”。
  - Codex 与 Claude 本地结构都不是正式公共 API，容错与降级是必需设计。
- **Alternatives considered**：
  - 读失败就回退到固定 `codex`：实现简单，但会制造系统性误导。
  - 读失败就清空 tray：过于激进，不利于稳定使用。
