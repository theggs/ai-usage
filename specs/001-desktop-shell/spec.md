# Feature Specification: Iteration 1 Desktop Shell

**Feature Branch**: `001-desktop-shell`  
**Created**: 2026-03-18  
**Status**: Draft  
**Input**: User description: "根据 doc/ai-usage-prd 中的产品描述，开始项目的第一个迭代，聚焦 Iteration 1 框架搭建：搭建可运行的跨平台空壳，包含 macOS 菜单栏与 Windows 托盘运行、弹出面板 UI 框架、设置页面骨架、开机自启、本地通知测试能力和 CI/CD 流水线，不包含任何真实 AI 服务的数据接入。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 从菜单栏或托盘快速查看应用状态 (Priority: P1)

作为重度使用 AI 编程工具的开发者，我希望应用在 macOS 菜单栏或 Windows 系统托盘常驻，并能点击后立即看到统一的状态面板，这样我可以确认应用已运行并为后续接入真实服务做好准备。

**Why this priority**: 这是整个产品的核心入口，也是后续所有额度展示能力的承载界面；如果没有常驻入口，产品的“零打断感知”目标无法验证。

**Independent Test**: 在 macOS 和 Windows 启动应用后，用户能看到常驻图标；点击图标后可在 1 秒内打开弹出面板，面板中展示预置的服务占位卡片、最近刷新时间占位信息、手动刷新入口和设置入口。

**Acceptance Scenarios**:

1. **Given** 用户首次启动应用，**When** 应用初始化完成，**Then** 应用显示为菜单栏或系统托盘常驻图标且不强制弹出主窗口。
2. **Given** 应用已常驻，**When** 用户点击图标，**Then** 系统展示弹出面板，包含服务列表占位区、刷新时间信息、手动刷新按钮和进入设置的入口。
3. **Given** 弹出面板已打开，**When** 用户看到服务列表，**Then** 每个占位服务都显示名称、图标占位、余量占位文案和状态占位，且明确标识当前为演示数据。

### User Story 2 - 在设置中完成基础偏好配置 (Priority: P1)

作为希望长期使用该工具的用户，我希望能进入设置页面并看到后续会使用到的关键配置项骨架，这样我能确认产品支持我的日常配置习惯并理解未来接入真实服务时的操作路径。

**Why this priority**: 设置入口决定产品是否具备持续可配置的基础，也是后续接入 API Key、刷新周期和语言设置的交互承载。

**Independent Test**: 用户可从弹出面板进入设置页面，并看到 API 凭证输入区、刷新间隔、开机自启、通知测试、菜单栏显示模式和语言选项等骨架；保存动作可反馈“已保存演示设置”。

**Acceptance Scenarios**:

1. **Given** 用户位于弹出面板，**When** 用户选择设置入口，**Then** 系统打开设置页面或设置视图，且不需要退出应用。
2. **Given** 用户进入设置页面，**When** 页面加载完成，**Then** 用户能看到 API 凭证管理、刷新间隔、开机自启、菜单栏显示模式、语言设置和通知测试入口的表单骨架。
3. **Given** 用户修改任意设置项，**When** 用户执行保存，**Then** 系统展示明确的保存反馈，并在下一次打开设置时保留这些演示设置值。

### User Story 3 - 验证桌面应用集成能力 (Priority: P2)

作为评估产品可行性的项目维护者，我希望第一迭代就验证开机自启、系统通知和安装包构建能力，这样可以降低后续跨平台桌面交付的风险。

**Why this priority**: 这些能力不直接呈现业务价值，但它们决定该产品能否成为“轻量、无感、原生”的桌面工具。

**Independent Test**: 用户可在设置中启用或关闭开机自启；可发送一条测试通知；项目代码库存在自动构建 macOS 与 Windows 安装包的持续集成流程定义。

**Acceptance Scenarios**:

1. **Given** 用户进入设置页面，**When** 用户打开或关闭开机自启，**Then** 系统保存偏好并反馈当前状态。
2. **Given** 用户位于设置页面，**When** 用户触发测试通知，**Then** 操作系统显示一条本地测试通知，文案可识别为 AIUsage 的演示通知。
3. **Given** 项目维护者查看代码库自动化流程，**When** 代码推送到主分支或手动触发构建，**Then** 流程会分别产出 macOS 与 Windows 的可分发安装包。

## Edge Cases

- 当用户在尚未配置任何真实服务时打开面板，系统仍需展示清晰的占位状态，而不是空白或报错。
- 当用户在系统不允许显示图标旁摘要的环境中运行应用时，系统仍需保留图标常驻与面板打开能力。
- 当测试通知因系统权限被拦截时，系统需明确告知用户通知未显示，并提示检查系统通知权限。
- 当用户频繁打开和关闭面板时，系统需保持响应稳定，不重复创建多个面板实例。
- 当用户切换语言后存在尚未本地化的占位文案时，系统需保持界面可理解，并优先回退到默认语言而非出现空字符串。
- 当自动构建流程在某一平台失败时，另一平台的构建结果仍应保留，便于独立排查。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须在 macOS 提供菜单栏常驻入口，并在 Windows 提供系统托盘常驻入口。
- **FR-002**: 系统必须在应用启动后默认静默常驻，不主动弹出主窗口打断用户当前工作。
- **FR-003**: 系统必须允许用户通过点击常驻图标打开和关闭统一的余量面板。
- **FR-004**: 系统必须在余量面板中展示至少一个服务列表占位区域，并为每个占位项提供服务名称、图标占位、余量占位、状态占位和最近刷新时间信息。
- **FR-005**: 系统必须在余量面板中提供手动刷新入口；在第一迭代中，该入口可刷新演示数据或刷新时间戳，但不得要求真实服务接入。
- **FR-006**: 系统必须在余量面板中提供进入设置的入口，并允许用户在不退出应用的情况下访问设置视图。
- **FR-007**: 系统必须提供设置页面骨架，至少覆盖 API 凭证管理、刷新间隔、菜单栏或托盘显示模式、开机自启、语言设置和通知测试入口。
- **FR-008**: 系统必须允许用户保存第一迭代提供的设置项，并在应用重启后恢复这些设置值。
- **FR-009**: 系统必须提供开机自启开关，并允许用户在设置中启用或禁用该能力。
- **FR-010**: 系统必须允许用户手动触发一条本地测试通知，以验证通知链路可用。
- **FR-011**: 系统必须明确标识第一迭代中的服务数据为演示内容，避免用户误认为已经接入真实额度。
- **FR-012**: 系统不得要求用户提供真实 API Key 才能完成第一迭代的全部主要流程验证。
- **FR-013**: 系统必须提供适用于 macOS 和 Windows 的自动化构建流程，并产出各自平台的安装包。
- **FR-014**: 系统必须确保第一迭代范围内不包含任何真实 AI 服务用量拉取、解析、汇总或告警逻辑。

### Key Entities *(include if feature involves data)*

- **Desktop Surface**: 桌面常驻入口及其关联弹出体验，包含图标、打开状态和当前显示摘要模式。
- **Panel Placeholder Item**: 面板中的演示服务项，包含服务名称、图标占位、余量占位、刷新时间和演示状态标识。
- **User Preference**: 用户在第一迭代可修改并持久保存的偏好设置，如刷新间隔、语言、显示模式和开机自启状态。
- **Notification Check**: 一次用户主动触发的测试通知记录，用于确认本地通知能力是否可用。
- **Build Artifact**: 按平台生成的安装包输出，用于验证桌面应用可被持续交付。

## Assumptions

- 第一迭代只需要证明跨平台桌面外壳、交互骨架和交付链路可行，不追求真实服务可用性。
- 面板中的服务列表可使用固定的演示服务项，以便提前验证布局、状态表达和后续扩展空间。
- 设置项在第一迭代中只需本地持久化，不要求与账户体系、云同步或团队共享打通。
- 测试通知只用于验证通知通路可达，不承担低额度告警策略验证。
- 自动构建流程以生成可安装产物为验收标准，不要求在第一迭代完成正式发布、签名或自动更新。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 90% 以上的内部验收尝试中，用户在启动应用后的 2 秒内能看到菜单栏或托盘图标常驻。
- **SC-002**: 90% 以上的内部验收尝试中，用户点击常驻图标后的 1 秒内能看到余量面板内容。
- **SC-003**: 100% 的第一迭代验收流程可在不输入真实 API Key 的前提下完成。
- **SC-004**: 至少 1 名验收人员在 macOS 和 1 名验收人员在 Windows 上分别成功完成“打开面板、进入设置、保存设置、触发测试通知”全流程。
- **SC-005**: 主分支上的自动化构建流程能够分别产出 macOS 和 Windows 的安装包结果，且任一平台成功率在连续 3 次验收构建中达到 100%。
