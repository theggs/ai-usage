# Feature Specification: UI/UX 视觉层级与交互优化

**Feature Branch**: `009-ui-ux-polish`
**Created**: 2026-03-23
**Status**: Draft
**Input**: User description: "基于真实 Tauri 窗口 UI/UX 审查（specs/008-ui-ux-review/review.md），优化面板视觉层级、设置页交互模式和微交互反馈，使产品更贴合 PRD 核心目标——打开弹窗 < 1 秒即可看到所有服务余量。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 一眼扫视余量状态 (Priority: P1)

用户点击菜单栏图标弹出面板后，不需要逐行阅读数字，仅通过进度条粗细、颜色、内嵌数字和卡片级视觉差异，即可在 1 秒内判断"哪个服务的哪个维度额度偏低"。

**Why this priority**: 这是 PRD 第一核心指标（"打开弹窗 < 1 秒即可看到所有服务余量"）。当前进度条太细（12px）、百分比数字太小且无色彩区分、卡片间无紧急状态差异，导致用户必须逐行细读，违背了产品设计初衷。

**Independent Test**: 可独立测试——只需改进面板卡片中的进度条和百分比数字的视觉呈现，不依赖设置页或其他功能。

**Acceptance Scenarios**:

1. **Given** 面板已打开且有多个服务的余量数据，**When** 用户看到面板，**Then** 进度条高度不低于 20px，百分比数字嵌入进度条内或以加粗加大字号显示，余量低于 50% 的数字使用对应警告色（amber/red）。
2. **Given** 某服务的某个维度余量低于 50%，**When** 面板展示该服务卡片，**Then** 该卡片通过左侧彩色边条或背景色变化与正常卡片产生视觉差异，用户无需阅读数字即可注意到异常卡片。
3. **Given** 所有服务余量均在 50% 以上，**When** 用户查看面板，**Then** 所有卡片视觉一致，无多余警告干扰。

---

### User Story 2 - 面板头部传达整体健康度 (Priority: P1)

用户打开面板后，在头部区域立即看到一行整体健康度摘要（如"所有服务正常"或"Claude Code 周额度偏低"），无需滚动到具体卡片即可判断是否需要关注。

**Why this priority**: 头部是面板中视觉权重最高的位置。当前显示的"额度面板"四个字零信息量，浪费了最宝贵的空间。对于 PRD "< 1 秒看到余量"的目标，健康度摘要是最直接的实现路径。

**Independent Test**: 可独立测试——只需修改面板头部文案逻辑，不影响卡片内容或设置页。

**Acceptance Scenarios**:

1. **Given** 所有服务的所有维度余量均大于 50%，**When** 面板打开，**Then** 头部显示正面状态信息（如"所有服务正常"），视觉上为绿色或中性色调。
2. **Given** 某服务的某维度余量低于 50%（但高于 20%），**When** 面板打开，**Then** 头部显示该服务和维度的警告摘要，视觉上为 amber 色调。
3. **Given** 某服务的某维度余量低于 20%，**When** 面板打开，**Then** 头部显示紧急警告摘要，视觉上为 red 色调。
4. **Given** 多个维度同时偏低，**When** 面板打开，**Then** 头部显示最紧急的那个维度信息，不堆叠多条告警。

---

### User Story 3 - 设置即时生效 (Priority: P1)

用户在设置页修改任何偏好设置后，更改立即生效，无需手动点击保存按钮。所有设置项使用统一的交互模式，用户不会困惑"改了生效了吗？"。

**Why this priority**: 当前 autostart toggle 即时生效但其他设置需手动保存，两种模式混用造成认知混乱。这是设置页最大的交互缺陷。

**Independent Test**: 可独立测试——修改某个设置（如语言），验证切换后立即生效而无需点击保存。

**Acceptance Scenarios**:

1. **Given** 用户在设置页，**When** 用户修改语言选择，**Then** 界面语言立即切换，无需额外操作。
2. **Given** 用户在设置页，**When** 用户修改刷新间隔，**Then** 新的刷新间隔立即应用到后台轮询。
3. **Given** 用户在设置页，**When** 用户修改托盘摘要规则、菜单栏服务等偏好，**Then** 更改立即生效。
4. **Given** 用户在设置页修改网络代理配置，**When** 输入代理 URL 后点击"应用"，**Then** 代理配置生效。代理是唯一保留显式确认按钮的设置项。
5. **Given** 即时生效模式下，**When** 用户误操作修改了设置，**Then** 用户可通过再次修改恢复，系统无需提供"撤销"功能。

---

### User Story 4 - 信息精简与一致性 (Priority: P2)

面板中的冗余和不一致信息被清理：时间格式统一使用相对时间、"上次刷新"不再每卡重复、徽章含义明确、维度标签对用户友好。用户的认知负担降低。

**Why this priority**: 这些是影响信息密度和阅读效率的细节问题。不修复不影响功能，但会持续拖累用户体验，尤其在未来接入更多服务后更明显。

**Independent Test**: 可独立测试——验证时间格式、刷新时间位置、徽章文案和维度标签是否符合预期。

**Acceptance Scenarios**:

1. **Given** 面板已打开，**When** 用户查看刷新时间，**Then** 显示为相对时间（如"3 分钟前"、"刚刚"），鼠标悬停时显示绝对时间。
2. **Given** 面板有多个服务卡片，**When** 所有服务的上次刷新时间相同，**Then** 只在面板底部显示一次全局刷新时间，卡片内不重复。
3. **Given** 面板已打开，**When** 用户查看额度维度标签，**Then** 显示用户友好的完整标签（如"5 小时窗口"、"每周额度"），而非缩写（"5h"、"week"）。
4. **Given** 服务已正常连接，**When** 用户查看卡片，**Then** 不显示冗余的"实时"徽章。仅在异常状态（离线、过期）时才显示状态徽章。

---

### User Story 5 - 服务拖拽排序 (Priority: P2)

用户在设置页通过拖拽操作调整服务在面板中的显示顺序，替代当前的上下文字按钮。

**Why this priority**: PRD 4.2 后期扩展明确提到"支持用户通过拖拽自定义服务显示顺序"。当前上下按钮与整体 UI 精致感不匹配，且随服务数量增加会越来越难用。

**Independent Test**: 可独立测试——进入设置页，拖拽排序某个服务，返回面板验证顺序是否更新。

**Acceptance Scenarios**:

1. **Given** 用户在设置页的服务排序区域，**When** 用户拖拽某个服务到新位置，**Then** 排序立即更新并持久化，返回面板后按新顺序显示。
2. **Given** 用户在设置页，**When** 查看服务排序区域，**Then** 每个服务项左侧显示拖拽手柄图标，不再有文字按钮。

---

### User Story 6 - 微交互反馈 (Priority: P2)

面板的关键操作（刷新、视图切换）提供清晰的视觉反馈，用户能确认操作已生效。

**Why this priority**: 微交互是"精致感"的重要组成部分，直接影响用户对产品品质的感知。当前刷新无动画、视图切换硬切，感觉粗糙。

**Independent Test**: 可独立测试——点击刷新按钮验证旋转动画、切换到设置页验证过渡动画。

**Acceptance Scenarios**:

1. **Given** 用户点击刷新按钮，**When** 数据正在刷新，**Then** 刷新图标显示旋转动画，刷新完成后恢复静止。
2. **Given** 用户在面板点击设置按钮，**When** 视图切换到设置页，**Then** 设置页以滑入动画出现（从右向左）。
3. **Given** 用户在设置页点击返回按钮，**When** 视图切换回面板，**Then** 面板以滑入动画出现（从左向右），返回按钮带有方向箭头图标。
4. **Given** 面板内容超出视窗高度，**When** 用户滚动内容，**Then** 顶部 sticky header 与内容之间出现分隔线或阴影指示。
5. **Given** 面板数据刷新导致进度条百分比变化，**When** 新数据渲染，**Then** 进度条宽度以平滑动画过渡到新值。

---

### User Story 7 - 设置页信息层级优化 (Priority: P3)

设置页内容按逻辑分组，关键设置在首屏可见，连接状态信息与用户设置在视觉上分离，低频功能降低视觉权重。

**Why this priority**: 当前设置页在 620px 窗口中只有前 5 项可见，保存按钮、代理、通知测试、Codex CLI 信息全在折叠区。信息层级重组能提升设置效率，但优先级低于面板核心体验。

**Independent Test**: 可独立测试——打开设置页，验证分组、首屏内容、视觉层级是否符合预期。

**Acceptance Scenarios**:

1. **Given** 用户打开设置页，**When** 查看首屏内容，**Then** 最常用的设置项（语言、自启动、摘要规则）在首屏可见，不需滚动。
2. **Given** 用户查看设置页，**When** 查看设置分组，**Then** 设置按逻辑分为多个组（如通用、显示、连接），组标题清晰。
3. **Given** 用户查看设置页底部，**When** 看到 Codex CLI 信息区域，**Then** 该区域使用浅色信息风格（而非深色卡片），明确传达这是"状态信息"而非可操作的设置。
4. **Given** 用户查看设置页，**When** 看到"通知测试"按钮，**Then** 该按钮视觉权重低于主操作按钮（outline/ghost 风格），位于页面底部。

---

### Edge Cases

- 所有服务均未连接时，面板头部健康度摘要如何显示？——显示引导信息（如"尚未连接任何服务"），而非空白或"所有服务正常"。
- 只有一个服务时，拖拽排序交互是否仍然展示？——展示拖拽手柄但不可拖拽（灰色），避免用户困惑为何看到手柄却无法操作。
- 刷新失败时，刷新按钮动画如何结束？——停止旋转，图标短暂变为错误状态（如红色），1 秒后恢复正常。
- 设置即时生效后用户立即关闭面板，设置是否已保存？——是，修改即持久化，不依赖面板保持打开。
- 相对时间的更新频率——"3 分钟前"不需要实时秒级更新。每 30 秒或下次数据刷新时更新即可。

## Requirements *(mandatory)*

### Functional Requirements

**面板 - 视觉层级**

- **FR-001**: 面板进度条高度必须不低于 20px，支持在条内显示百分比数字。
- **FR-002**: 余量低于 50% 的百分比数字和进度条必须使用 amber 警告色，低于 20% 使用 red 紧急色。
- **FR-003**: 面板头部必须显示整体健康度摘要，反映所有服务中最紧急的余量状态。
- **FR-004**: 当某服务存在低于 50% 的维度时，该服务卡片必须通过视觉差异（彩色边条或背景变化）与正常卡片区分。

**面板 - 信息精简**

- **FR-005**: 面板中的时间显示必须统一使用相对时间格式（"刚刚"、"N 分钟前"、"N 小时前"），鼠标悬停显示绝对时间。
- **FR-006**: 当所有服务的刷新时间相同时，"上次刷新"必须只在面板底部全局显示一次，不在每张卡片中重复。
- **FR-007**: 额度维度标签必须使用用户友好的完整名称（如"5 小时窗口"、"每周额度"），取代缩写。
- **FR-008**: 连接正常的服务卡片不再显示"实时"徽章。仅在异常状态（离线、凭证过期、会话恢复中）时显示状态徽章。

**设置页**

- **FR-009**: 除网络代理外，所有偏好设置必须在修改后立即生效并持久化，无需手动点击保存按钮。
- **FR-010**: 通用"保存偏好"按钮必须移除。网络代理设置保留单独的"应用"按钮。
- **FR-011**: 服务排序必须支持拖拽操作，排序结果立即生效并持久化。原有的上下文字按钮必须移除。

**微交互**

- **FR-012**: 刷新按钮在数据刷新期间必须显示旋转动画。
- **FR-013**: 面板与设置之间的视图切换必须有滑动过渡动画。
- **FR-014**: 设置页"返回"按钮必须包含方向箭头图标。
- **FR-015**: 面板内容滚动时，sticky header 底部必须出现分隔线或阴影。
- **FR-016**: 进度条宽度变化时必须有平滑过渡动画。

**设置页 - 信息层级**

- **FR-017**: 设置项必须按逻辑分组，最常用设置项在 620px 窗口首屏可见。
- **FR-018**: Codex CLI 信息区域必须使用浅色信息风格（与深色卡片风格不同），视觉上与可操作设置项区分。
- **FR-019**: "通知测试"按钮必须使用 ghost/outline 风格，视觉权重低于主操作按钮。

### Assumptions

- 产品面向 macOS 菜单栏，窗口固定 360×620px，所有布局优化基于此尺寸。
- 当前支持 2 个服务（Claude Code、Codex），设计应考虑未来扩展到 4-5 个服务。
- 拖拽排序仅需支持鼠标操作（桌面应用），无需考虑触屏。
- 即时生效不需要"撤销"功能，用户可通过再次修改恢复。
- 相对时间不需要实时秒级刷新，每 30 秒或数据刷新时更新即可。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户打开面板后，能在 1 秒内识别出余量最低的服务和维度（通过卡片视觉差异和头部摘要）。
- **SC-002**: 面板中不存在同一信息在两个位置重复展示的情况（如"上次刷新"只出现一次）。
- **SC-003**: 面板中所有时间信息使用相对时间格式，格式一致性 100%。
- **SC-004**: 用户修改设置后，0 秒等待即生效（无需手动保存），覆盖所有非代理设置项。
- **SC-005**: 面板中的每次状态变化（刷新、视图切换、进度条更新）都有对应的动画反馈，无硬切或静默状态。
- **SC-006**: 设置页在 620px 高的窗口中，最常用的 3 项设置在首屏可见，无需滚动。
- **SC-007**: 自动化 E2E 截图审查中，进度条内可见嵌入的百分比数字，低余量卡片与正常卡片间存在可识别的视觉差异。
