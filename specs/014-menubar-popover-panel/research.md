# Research：菜单栏吸附式弹出面板

**Feature**：014-menubar-popover-panel | **Date**：2026-03-25

## 决策 1：弹出定位由 Rust 宿主层统一负责

- **Decision**：主面板的弹出位置、边缘收敛、多显示器选择和异常回退都在 Tauri 宿主层完成，tray 点击不再只是简单 `show()/hide()`。
- **Rationale**：菜单栏图标几何信息、显示器工作区和窗口最终位置都是原生窗口能力。把这些逻辑放在 Rust 宿主层可以避免前端猜测屏幕坐标，也更容易做纯函数测试和真实桌面回归。
- **Alternatives considered**：把位置计算放到 React 侧被否决，因为前端无法稳定拿到 tray 图标几何和工作区边界；继续沿用当前 `show()/hide()` 被否决，因为无法满足吸附式弹出的 spec。

## 决策 2：优先用“无装饰单一面板”达成视觉目标，而不是默认引入真正透明窗口

- **Decision**：默认方案使用无装饰窗口 + 单一白色面板本体来实现视觉结果，不把真正透明窗口作为基础方案。
- **Rationale**：规格要求的是“视觉上只保留单一面板本体”，而不是必须使用系统级透明窗口。现有灰色外衬底主要来自前端壳层和外层容器，先收敛壳层就能满足目标，同时避免 macOS 私有透明 API 的额外风险和验证成本。
- **Alternatives considered**：直接启用透明窗口被否决为默认路径，因为这会引入更高的平台风险；保留外层灰色/渐变背景只弱化一点样式被否决，因为无法满足 `FR-006a`。

## 决策 3：横向对齐采用“图标中心线优先，越界再收敛”

- **Decision**：在有足够空间时，主面板以菜单栏图标水平中心线为基准在其下方弹出；如果按此规则即将越界，再向当前显示器工作区内收敛。
- **Rationale**：这与澄清结论一致，也最符合用户对菜单栏工具“从图标下面弹出”的直觉，同时能兼容左、中、右不同位置的图标。
- **Alternatives considered**：固定左对齐或右对齐被否决，因为在居中或另一侧边缘时观感较差；完全采用统一固定偏移被否决，因为会削弱“跟随当前图标”的感知。

## 决策 4：高度采用内容自适应 + 工作区上限 + 内部滚动

- **Decision**：主面板在内容较少时自适应收紧，在内容超过允许上限后停止继续增高，并在面板内部滚动访问超出内容。
- **Rationale**：这能同时满足紧凑感、菜单栏工具视觉节奏和工作区安全约束，也能兼顾 panel 与 settings 两个视图的高度差异。
- **Alternatives considered**：固定接近当前大窗口高度被否决，因为不符合菜单栏面板气质；完全无限自适应被否决，因为设置页内容可能超出屏幕可用高度。

## 决策 5：定位失败的回退顺序显式化

- **Decision**：显示主面板时的回退顺序为：`tray 图标几何信息 -> 上一次成功弹出位置 -> 当前显示器顶部安全默认位`。
- **Rationale**：规格已明确“无法获取本次菜单栏图标精确位置时回退到上一次成功位置”。为了消除首次异常时仍无位置可用的技术空洞，计划额外定义顶部安全默认位作为最终兜底，但这只在前两者都不可用时触发。
- **Alternatives considered**：回退到屏幕中央被否决，因为会显得像普通窗口；直接不弹出被否决，因为损害可用性；每次异常都沿用随机系统默认位置被否决，因为不可预测。

## 决策 6：重新打开时回到 panel 视图，而不是保留次级页

- **Decision**：主面板从隐藏状态重新显示时，前端壳层默认回到主额度面板视图，并清理与次级页面/浮层相关的瞬态状态。
- **Rationale**：这与 `FR-014` 一致，也符合菜单栏工具“快速看一眼”的核心心智。让用户重新打开时仍停留在 settings 会把菜单栏弹出面板变成“迷你独立应用”。
- **Alternatives considered**：保留上一次视图被否决，因为会增加认知切换成本；每次重新挂载整个应用被否决，因为会放大状态闪烁和加载复杂度。

## 决策 7：本特性不新增手动拖动能力

- **Decision**：这次不为主面板新增用户可见的手动拖动能力，吸附式定位仍然是权威行为。
- **Rationale**：拖动与“从图标下方弹出”的主目标冲突，也会引入拖后位置记忆、再次打开是否重置、与边缘收敛冲突等额外复杂度。当前 spec 已能完整定义目标价值，不需要靠拖动补位。
- **Alternatives considered**：增加自定义 drag region 被否决，因为会显著抬高实现和测试复杂度；继续依赖传统标题栏拖动被否决，因为本特性目标正是去除标题栏和传统窗口感。

## 决策 8：完成门槛必须包含真实桌面验证

- **Decision**：本特性的完成标准必须同时包含宿主单测、React/RTL 回归以及真实 Tauri 窗口的 E2E/截图验证。
- **Rationale**：定位、边缘收敛、无灰色外衬底、重新打开视图与失焦收起都不能只靠 JSDOM 证明。项目工作流也明确要求桌面 UI 变更必须经过真实运行时验证。
- **Alternatives considered**：只做前端测试被否决，因为无法覆盖宿主定位与真实窗口行为；只做人工目测被否决，因为缺乏可重复的回归门槛。
