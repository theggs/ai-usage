# 研究记录：UI/UX 审查遗留项完成

**阶段**：0 —— 研究与决策记录  
**特性**：010-ui-ux-completion  
**日期**：2026-03-23

> 变更说明（回退）：
> tray warning / danger 黄红状态图标需求已放弃。下述研究结论已按“菜单栏仅保留标题与 tooltip 同步，图标保持默认应用图标”回退。

## 决策 1 —— 菜单栏刷新一致性必须由宿主负责

**决策**：每次手动刷新与自动刷新成功后，都由 Rust 宿主重新计算并应用 tray 标题与 tooltip，且使用与面板相同的一组规范 `PanelPlaceholderItem` 数据。tray 图标保持默认应用图标。

**理由**：这次问题的本质是真实状态表达错误，而不是展示偏好问题。tray 标题/tooltip 本来就由宿主负责，因此最安全的修复方式，是保证所有“刷新成功”路径最终都汇聚到同一套 tray 表面重算逻辑上。这样可以避免在 React 中维护第二份状态来源，并让面板和菜单栏始终绑定到同一组规范数据。

**备选方案**：
- 由 React 在面板刷新后主动推送 tray 更新：否决，因为 tray 原生行为应属于宿主边界
- 仅在偏好变更和手动刷新时更新 tray：否决，因为自动刷新仍会不一致
- 将 tray 文本独立缓存：否决，因为会引入漂移风险

---

## 决策 2 —— 强调条应是装饰元素，而不是结构元素

**决策**：将 warning/danger 强调条实现为卡片容器内部的绝对定位装饰元素，并由卡片正文为其预留内边距。

**理由**：当前顶部空白的根因是强调条处于正常文档流中。把它改为装饰元素后，可以在不改变卡片语义结构的前提下，让强调条覆盖整张卡片高度，且不再占用额外垂直空间。这是直接对应 #29 的最小修复方案。

**备选方案**：
- 保留现有元素，仅缩小高度或 margin：否决，因为依旧脆弱，且仍参与布局
- 将强调条只放在 header 行内：否决，因为规格要求强调条沿卡片全高延展
- 改为单纯的 `border-left`：否决，因为 warning/danger 需要比普通边框更强的视觉权重和圆角表现

---

## 决策 3 —— 服务识别回退为纯文本；菜单栏图标回退为默认应用图标

**决策**：面板服务卡片不再交付 Codex / Claude Code 的品牌图标、首字母头像或文字徽章，统一使用服务名文本作为识别元素。tray 图标不再交付 warning / danger 变体，统一回退为默认应用图标。

**理由**：当前服务数量有限，服务名文本本身已足够识别；继续引入图标或文字徽章只会与标题重复，并且对外发布时还会带来品牌资产与一致性风险。对 tray 而言，黄红状态变体验证后未达到预期，继续交付会引入视觉风险，因此回退为默认应用图标更稳妥。

**备选方案**：
- 远程加载 logo：否决，因为会引入网络依赖与缓存复杂度
- 为所有服务继续提供图形徽章：否决，因为会与服务名文本重复，且难以在不使用官方品牌资产的前提下达到理想审美
- 运行时动态绘制 tray 图标：否决，因为该需求已放弃，不再投入

---

## 决策 4 —— 排序必须从 HTML5 DnD 迁移到 Pointer Events

**决策**：将当前基于 `draggable` / `onDrop` 的设置列表排序，替换为基于 Pointer Events 的可排序交互，并由纯函数 helper 支撑排序逻辑。

**理由**：审查报告已明确指出 WKWebView 下的可靠性问题，而当前实现正是 HTML5 drag/drop。Pointer Events 在 Tauri WebView 中对鼠标与触控板行为更可预测，同时无需引入新依赖，也更适合当前这种短列表桌面交互。

**备选方案**：
- 保留 HTML5 drag/drop 并补边角情况：否决，因为平台兼容性本身就是核心问题
- 引入 `dnd-kit`：否决，因为列表很短，交互边界明确，依赖成本不值得
- 回退为上下移动按钮：否决，因为不满足需求里要求的拖拽排序模型

---

## 决策 5 —— 首次引导复用现有偏好存储

**决策**：在现有 `UserPreferences` 中新增一个可选的引导关闭时间戳/标记，以便用户跳过一次首次引导后，后续看到的是按服务拆分的空状态占位引导。

**理由**：规格明确区分了“首次打开时覆盖整个面板的引导”和“后续各服务自己的未连接占位卡”，并要求支持跳过。这种行为只需要一个持久化比特位，不值得为此增加新的存储机制。扩展现有偏好文件可以保持本地优先，也能通过默认缺失值实现向后兼容。

**备选方案**：
- 在服务未连接前每次启动都展示首次引导：否决，因为不符合“可跳过”要求
- 只放在 React 内存状态里：否决，因为重启应用后会再次出现
- 新建独立 onboarding store：否决，因为仓库已有约束要求优先复用现有偏好层

---

## 决策 6 —— 设置页状态必须按服务归一化，而不是只展示 Codex

**决策**：设置页状态区域按服务渲染状态卡片，分别从 Codex 与 Claude Code 当前 panel state 派生连接状态、数据来源以及会话/快照细节。

**理由**：当前只展示 Codex 状态，会在多服务场景下违反“真实用户状态”原则。将状态卡片按服务归一化后，设置页才能与实际支持的集成能力保持一致，也能避免在单服务场景中出现误导性的“无活动会话”文案。

**备选方案**：
- 保留一个通用状态卡并附加额外文本：否决，因为无法明确状态属于哪个服务
- 只展示当前菜单栏选中的服务：否决，因为设置页本来就是用户检查所有服务配置的地方
- 把状态详情移出设置页：否决，因为规格明确要求在设置页补全状态信息

---

## 决策 7 —— 英文健康摘要应靠缩短文案解决，而不是依赖省略号

**决策**：将英文 header 健康摘要改为更短的服务聚焦短句，例如 `Codex low on 5h quota` / `Claude Code weekly quota low`，并避免把截断当作主要解法。

**理由**：面板宽度是固定的，如果仍依赖 `truncate`，只是把原问题换一种方式延续。更短的文案可以保留最关键的信息，也能和中文里的紧急度语义保持一致，而不需要额外占据更多横向空间。

**备选方案**：
- 通过缩小按钮给 header 文案腾空间：否决，因为会破坏统一的 header 结构
- 允许摘要换成两行：否决，因为会破坏紧凑面板节奏
- 保持当前文案并允许 ellipsis：否决，因为关键的服务名与严重程度仍会被隐藏
