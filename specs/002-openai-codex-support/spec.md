## User Scenarios & Testing *(mandatory)*

### User Story 1 - 在设置页理解本地 Codex CLI 同步方式 (Priority: P1)

作为想确认桌面应用是否已经接上真实 Codex 额度的开发者，我希望设置页直接说明第二迭代依赖本地活跃 Codex CLI 会话，而不是要求我手动填写账户或凭证，这样我能快速理解正确的使用路径。

**Why this priority**: 如果设置页继续把“账户/凭证录入”作为主流程，用户会误以为第二迭代依赖手工配置，而不是本地 CLI 状态。

**Independent Test**: 用户进入设置页后，可以看到当前本地 Codex CLI 同步状态、数据来源与使用提示；页面不再要求新增、启用或删除手工账户条目。

**Acceptance Scenarios**:

1. **Given** 用户进入设置页，**When** 当前本地 Codex CLI 已可读取，**Then** 页面展示活跃会话状态与数据来源说明。
2. **Given** 用户进入设置页，**When** 当前本地 Codex CLI 尚未连通，**Then** 页面展示明确的待同步或未连接提示。
3. **Given** 用户查看设置页，**When** 用户寻找配置入口，**Then** 页面不会要求手工录入 Codex 账户、凭证标签或组织标签。
4. **Given** 后续仍需要扩展多账户能力，**When** 维护者查看规格，**Then** 能看到这类元数据仅作为未来能力预留，不是第二迭代主流程。
5. **Given** 用户打开设置页，**When** 用户调整托盘摘要规则并保存，**Then** 新规则会立即影响托盘摘要预览并在应用重启后继续生效。

### User Story 2 - 在面板中看到 Codex 额度状态 (Priority: P1)

作为需要快速确认额度状态的用户，我希望在面板里立即看到当前本地 Codex CLI 的同步状态，以及当前活跃 Codex 会话的各额度维度状态，这样我能判断第二迭代是否已经具备真实监控价值，而不是只看到第一迭代的纯演示壳。

**Why this priority**: 第二迭代首先要让面板承载真实 Codex 额度状态，否则设置和面板仍然彼此割裂。

**Independent Test**: 当存在一个可读取的本地活跃会话后，回到面板能看到该活跃会话的额度状态；未连通时显示清晰的待同步、未连接或失败提示。

**Acceptance Scenarios**:

1. **Given** 用户当前本地 Codex CLI 尚未连通，**When** 用户打开面板，**Then** 面板明确提示当前同步仍待建立，而不是提示用户去新增账户。
2. **Given** 本地活跃 Codex 会话可读取，**When** 用户打开面板，**Then** 面板展示该会话可见的额度维度状态。
3. **Given** 用户从设置页返回面板，**When** 本地会话状态已变化，**Then** 面板中的同步状态摘要即时反映最新状态。
4. **Given** 当前没有可读取的本地会话，**When** 用户打开面板，**Then** 面板需明确这是 CLI 同步问题，而不是账户配置缺失。
5. **Given** 当前存在多个额度维度，**When** 用户把托盘摘要规则设为最低余量、5 小时窗口、7 天/周窗口或多维摘要，**Then** 托盘摘要必须按该规则显示对应值。

### User Story 3 - 为真实 Codex 额度接入建立正确的业务边界 (Priority: P2)

作为项目维护者，我希望第二迭代在代码和文档层明确真实 Codex 额度接入的宿主边界、存储边界和同步约束，这样后续接入不会把本地会话信息暴露到前端之外，也不会基于错误假设返工。

**Why this priority**: Codex 的额度查看方式与通用 API 用量统计并不相同，而且第二迭代已明确优先围绕本地 CLI 可见额度来做；如果一开始边界定义错误，后续安全和实现都会被拖慢。

**Independent Test**: 仓库中存在第二迭代规格、研究结论和任务拆解，且前端代码不直接承担真实 Codex 凭证出网职责，也不直接解析终端会话信息。

**Acceptance Scenarios**:

1. **Given** 维护者查看第二迭代文档，**When** 阅读研究与计划，**Then** 能看到 Codex CLI 结构化额度读取路径及其对实现边界的影响。
2. **Given** 维护者检查前端实现，**When** 查找 Codex 实时同步代码，**Then** 不会发现前端直接持有并解析真实 CLI 会话信息的实现。
3. **Given** 后续要继续开发真实同步，**When** 开发者阅读任务拆解，**Then** 能按账户管理、宿主同步、面板展示三个方向继续推进。

## Edge Cases

- 当不同 Codex 额度维度的重置周期不同步时，面板需按维度分别展示，而不是合并成单一百分比。
- 当用户选择某个托盘摘要维度，但当前快照里不存在该维度时，系统需退回到“最低余量”而不是显示空白或过期值。
- 当本地未登录 Codex CLI 或当前环境无法读取结构化额度信息时，系统需显示明确的未连接状态，而不是沿用旧额度数据。
- 在真实 Codex 同步尚未接入前，界面需明确标识当前是“接入准备中”，避免用户误认为已经读取到真实额度。
- 即使底层保留未来多账户元数据能力，当前界面也不得把它暴露成必填配置步骤。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须在设置页明确说明第二迭代的真实数据来源是本地 Codex CLI 会话，而不是手工账户或凭证输入。
- **FR-002**: 设置页不得把 Codex 账户、凭证标签或组织标签作为当前用户完成同步的必填步骤。
- **FR-003**: 系统必须在应用启动后直接尝试读取本地 Codex CLI 状态，而不要求先完成账户配置。
- **FR-003a**: 第二迭代必须通过宿主层调用本地 Codex CLI 的结构化 rate-limit 读取路径，而不是要求前端解析 `/status` 文本。
- **FR-004**: 系统必须在面板中显示当前本地 Codex CLI 的同步状态。
- **FR-005**: 当本地 Codex CLI 同步成功时，系统必须展示一个或多个额度维度的当前状态。
- **FR-006**: 当本地 Codex CLI 尚未连通或不可读取时，系统必须在面板中展示明确的待同步、未连接或失败提示。
- **FR-007**: 当同步成功时，系统必须围绕当前活跃会话展示一个或多个额度维度的当前状态。
- **FR-008**: 当不同额度维度具有不同重置周期时，系统必须分别展示这些维度，而不是压平成单一指标。
- **FR-009**: 第二迭代必须优先围绕本地 Codex CLI 的结构化额度信息构建同步能力。
- **FR-009a**: 第二迭代必须支持托盘摘要规则设置，至少包含 `最低余量`、`5h 窗口`、`7d/周窗口`、`多维摘要`、`仅图标` 五种模式。
- **FR-009b**: 第二迭代必须将 `最低余量` 作为默认托盘摘要策略。
- **FR-009c**: 用户保存托盘摘要规则后，系统必须立即更新托盘摘要，并在应用重启后恢复该规则。
- **FR-010**: 第二迭代在真实同步层面必须先支持单个本地活跃 Codex 会话。
- **FR-011**: 如果底层保留多账户预留元数据，第二迭代界面也不得把这些预留项当成当前用户的主操作对象。
- **FR-012**: 当本地无法提供 Codex CLI 额度信息时，系统必须展示明确的未连接或待登录状态。
- **FR-013**: 在真实额度同步尚未完成前，系统必须在界面中明确标识当前同步链路仍待接入。
- **FR-014**: 第二迭代的前端实现不得直接承担真实 Codex 凭证出网职责，也不得直接承担 CLI 输出解析职责。
- **FR-015**: 仓库必须补充第二迭代的研究、计划和任务文档，以记录 Codex 额度查看约束和后续实现顺序。

### Key Entities *(include if feature involves data)*

- **Reserved Codex Account Metadata**: 底层可选保留的未来扩展元数据，不属于第二迭代用户主流程。
- **Active Codex Session**: 当前设备上可被读取真实额度信息的单个本地 Codex 会话，与预留账户配置分开管理。
- **Codex Limit Dimension**: 某个 Codex 账户下的单一额度维度，例如 5 小时窗口或周周期额度，包含当前状态与重置提示。
- **Codex Sync Summary**: 面板中汇总展示的本地 Codex CLI 同步状态，包含当前同步准备状态、数据来源与活跃会话信息。
- **CLI Limit Snapshot**: 一次从本地 Codex CLI 结构化额度接口中读取到的额度快照，包含账户、维度、剩余状态和读取时间。
- **Tray Summary Rule**: 用户为菜单栏/托盘摘要选择的显示规则，可为最低余量、指定窗口、多维摘要或仅图标，并需持久化。
- **Usage Sync Boundary**: 第二迭代中前端、本地持久化和未来宿主同步之间的职责边界说明。

## Assumptions

- 第二迭代主监控目标是 Codex usage limits，而不是更宽泛的 OpenAI API usage / costs。
- 第二迭代的真实同步范围先限定为单个本地活跃 Codex 会话，多账户真实切换留待后续迭代。
- 当前真实读取实现优先使用本地 Codex CLI 的结构化 rate-limit 响应，而不是前端或用户手工处理 CLI 文本。
- 即使底层保留未来多账户元数据，也不会在第二迭代界面中要求用户手工录入这些内容。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户可以在 30 秒内理解第二迭代依赖本地 Codex CLI，同步路径不需要手工账户录入。
- **SC-002**: 100% 的面板打开场景都能看到 Codex CLI 同步状态摘要，不再只有第一迭代的纯演示语义。
- **SC-003**: 用户修改托盘摘要规则后，无需重启即可看到新摘要，并且重启应用后仍保持该规则。
- **SC-004**: 对于已同步成功的单个活跃 Codex 会话，用户在面板中能看到至少一个真实额度维度及其当前状态。
- **SC-005**: 维护者可通过第二迭代文档在 5 分钟内理解真实 Codex 额度接入为什么必须落在宿主安全边界内。
