# Research: 优惠活动提示

**Feature**: 013-promotion-status  
**Date**: 2026-03-25  
**Spec**: [spec.md](./spec.md)

## Decision 1: 用源码内独立活动目录作为促销活动的权威来源

**Decision**: 促销活动定义、活动历史和时间规则统一保存在源码内的独立活动目录中，由版本控制承担留痕职责；运行时不抓取帮助中心页面，也不依赖远端配置。

**Rationale**:
- 符合本项目 local-first 与无云依赖原则
- 活动结束后仍可保留历史记录与规则版本
- 避免运行时抓取页面带来的脆弱性、时效性抖动和误判

**Alternatives considered**:
- 运行时抓取官方帮助中心页面：否决，页面结构不稳定，且会让 UI 真值依赖远端可用性
- 引入远端配置或后台管理：否决，超出当前项目架构与需求范围

## Decision 2: 促销逻辑使用前端纯解析器，不下沉到宿主层

**Decision**: 新增前端独立促销模块，使用纯函数根据活动目录、当前时间、已知服务可见性和资格信息生成展示结果；不新增 Rust 宿主命令。

**Rationale**:
- 本特性不需要新增敏感数据路径或系统能力
- 促销判断是纯本地派生逻辑，放在前端更容易测试和迭代
- 避免为纯 UI 能力扩张 host-to-UI 边界

**Alternatives considered**:
- 在 Rust 宿主层生成促销状态：否决，增加跨层复杂度，但没有带来新的安全收益
- 直接在 `AppShell` 里写分支逻辑：否决，无法满足“独立模块管理”和历史可追溯

## Decision 3: 头部采用“两级信息 + 默认胶囊 + 同区域完整浮层”

**Decision**: 头部左侧维持两级信息结构：主行保留现有头部主信息语义，副行新增促销提示；默认只显示“有优惠/待确认”的服务状态，并使用“服务图标 + 极短状态文案”的并列小胶囊表达。完整信息不再通过显式 `...` 入口切换，而是在主界面同一区域内以锚定浮层方式查看全部服务状态。

**Rationale**:
- 满足“标题下方小字提示”的产品要求
- 胶囊表达比纯文本串接更利于一眼扫读与服务比较
- 去掉 `...` 后可把有限横向空间留给真正有价值的状态信息

**Alternatives considered**:
- 默认始终展示所有服务：否决，随着服务增多头部会迅速失去可读性
- 把完整信息放到设置页：否决，打断了“快速选择该用哪个服务”的核心场景
- 把促销提示做成独立卡片：否决，层级过重，违背低打扰目标

## Decision 4: 完整浮层采用“hover/focus 预览 + click 稳定展开 + 外部点击/Esc 关闭”

**Decision**: hover 与 focus 用于临时预览完整状态；click 用于稳定展开完整浮层；稳定展开后的统一关闭方式是点击浮层外区域或按 `Esc`。

**Rationale**:
- 菜单栏面板空间紧，hover-only 容易因鼠标移入移出造成抖动
- 只保留 click toggle 会牺牲快速扫读能力
- 同时支持 hover/focus/click 更利于键盘可达性与真实壳层稳定性验证

**Alternatives considered**:
- 只靠 hover 展开：否决，发现性与可访问性不足
- click 与 hover 完全等价：否决，无法提供稳定展开态
- click 后再次点同一行关闭：否决，退出路径不如“外部点击/Esc”自然且一致

## Decision 5: 未知资格单独建模为中性状态，不隐藏也不冒进

**Decision**: 当用户计划未知、缺失或无法可靠判断时，该服务继续显示，但状态固定为“优惠资格待确认”，不得判定为“正在优惠时段”或“明确不适用”。

**Rationale**:
- 保持 truthful state
- 满足“帮助用户比较服务”的目标，不因信息不足把服务直接抹掉
- 比模糊地直接给出肯定性优惠判断更诚实

**Alternatives considered**:
- 隐藏该服务：否决，用户会误以为该服务没有促销相关信息
- 直接按最宽松规则显示有活动：否决，误导风险过高

## Decision 6: 时间规则用内建时间能力处理，不新增日期库

**Decision**: 促销时间窗口与时区转换优先使用现有 JavaScript 时间能力与受控样例测试，不新增运行时日期库。

**Rationale**:
- 当前规则规模小，主要是固定区间、工作日和少量官方时区
- 减少依赖与维护面
- 通过固定样例测试足以覆盖关键风险

**Alternatives considered**:
- 新增日期时间库：否决，当前复杂度不足以 justify 新依赖
- 只按本地时间字符串硬编码判断：否决，难以正确覆盖时区与工作日边界

## Decision 7: 验证必须覆盖真实壳层宽度，而不只靠单元测试

**Decision**: 在逻辑测试之外，必须使用真实 Tauri 壳层截图或 E2E 路径验证头部一行促销提示、胶囊表达、完整浮层稳定性，以及 click 展开后的关闭路径。

**Rationale**:
- 头部是固定宽度、高密度区域，JSDOM 不能证明真实可读性
- 浮层的 hover/click 行为高度依赖真实壳层中的鼠标、焦点和布局表现
- 仓库的 spec-kit workflow 已明确要求对桌面 UI 做真实运行态验证

**Alternatives considered**:
- 仅靠 Vitest/RTL：否决，无法证明一行聚合提示与浮层交互在真实面板里成立

## Decision 8: 完整浮层使用“两行信息块”，并把 Claude 时间信息转为“活动日期范围 + 用户本地优惠窗口”

**Decision**: 完整浮层中的每个服务项不再沿用默认胶囊的极简表达，而是使用“两行信息块”：第一行展示“图标 + 服务名 + 结论 + 可选优惠幅度”，第二行展示时间信息或时段说明。其中 Claude Code 的第二行统一显示“活动日期范围 + 用户本地优惠窗口”，Codex 的第二行统一显示“当前持续优惠时段”。

**Rationale**:
- 完整浮层已经脱离默认头部副行的极限紧凑约束，应该承载更多真正有助于选择服务的信息
- 用两行信息块能在狭窄菜单栏面板中保持更好的扫读节奏，避免把所有字段硬塞成一行长文本
- Claude 的官方规则包含 PT 日期范围与 ET 工作日窗口，若直接暴露原始规则会迫使用户自行换算；转为“活动日期范围 + 用户本地优惠窗口”更符合行动导向
- Codex 当前按连续优惠时段处理，用“当前持续优惠时段”比 `00:00-23:59` 更自然，也更少机械感

**Alternatives considered**:
- 继续在浮层中沿用紧凑胶囊：否决，信息密度不足，无法表达服务名、结论、幅度与时间语义
- 每个服务项塞成单行长文本：否决，在真实面板宽度下更容易换行难看、降低可读性
- Claude 第二行只显示本地时间戳或只显示日期范围：否决，都不足以帮助用户判断“我本地什么时候算优惠”
