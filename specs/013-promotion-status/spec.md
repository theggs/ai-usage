# Feature Specification: 优惠活动提示

**Feature Branch**: `013-promotion-status`  
**Created**: 2026-03-24  
**Status**: Draft  
**Input**: User description: "增加一个优惠活动提示的功能。在主界面的标题下方以小字展示当前时间是否处于优惠时段；促销活动需用独立模块管理，面向未来促销开展/结束变化和历史记录保留；先覆盖当前进行中的 Claude March 2026 usage promotion 与 Codex limited-time promotion；输出仅包含产品设计说明，不包含代码实现细节。"

## Product Principles

- 优先回答用户真正关心的问题：现在是否值得集中使用某个 AI 服务。
- 优先真实而非“看起来聪明”：只有在活动规则公开且可判断时，才显示“当前处于优惠时段”。
- 优先低打扰：优惠提示只占用标题下方一行小字，不与额度主信息争夺注意力。
- 优先可演进：优惠活动是会变化、会结束、会复用命名的运营对象，必须作为独立产品域管理，而不是写死在单次版本里。
- 优先可追溯：活动结束后仍保留历史记录、活动来源与生效区间，避免后续产品判断失真。

## Clarifications

### Session 2026-03-24

- Q: 当多个服务同时存在活动时，标题下方应优先显示哪个服务的优惠信息？ → A: 应显示所有服务的可用优惠信息，以提供给用户信息来选择使用哪个服务；“提供给用户信息选择使用哪个服务”就是该功能目标的一部分。
- Q: 在需要同时显示所有服务优惠信息的前提下，标题下方提示区域应采用什么信息密度？ → A: 保持标题下方一行小字，用紧凑格式同时显示所有服务状态。
- Q: 当用户计划未知、缺失或无法可靠判断时，该服务的优惠状态应如何展示？ → A: 该服务仍应显示，但需标记为“优惠资格待确认”等保守中性状态。
- Q: 默认情况下，是否要显示所有已支持服务的状态，还是只显示当前有优惠的服务？ → A: 默认只显示当前有优惠或待确认的服务，但用户可切换查看全部服务状态。
- Q: 标题下方的促销信息是否要继续保留显式文字前缀“促销信息：”？ → A: 不保留固定可见前缀；该行本身就是促销信息区，应把空间优先留给各服务状态本身。
- Q: 标题下方提示区域的最终展示形式应是什么？ → A: 采用“服务图标 + 极短状态文案”的并列小胶囊；颜色可以辅助传达状态，但不能脱离图标和文案单独承载语义。
- Q: “查看全部”能力应如何进入？ → A: 不再使用显式 `...` 入口；默认通过悬停在促销信息行上预览全部服务状态。
- Q: hover 是否可以作为唯一的“查看全部”进入方式？ → A: 不可以；除 hover 外，还必须提供 focus/click 的等效进入方式，以保证可发现性、稳定性与可访问性。
- Q: 用户进入“查看全部”状态后，全部服务状态应显示在哪里？ → A: 仍锚定在标题下方同一块促销信息区域上，以浮层形式展开全部服务状态，而不是跳到独立区域。
- Q: 当用户通过 click 打开完整浮层后，浮层应如何关闭？ → A: hover 仅用于临时预览；click 用于稳定展开，随后应通过点击外部区域或按 `Esc` 关闭。
- Q: 展开的优惠活动详情应采用什么信息结构？ → A: 每个服务使用“两行信息块”，第一行展示“图标 + 服务名 + 结论 + 幅度”，第二行展示时间信息；Claude Code 第二行初始表述为 `2026.03.13-2026.03.28 · 按用户当前时区标出的时间范围`，该表述已被 2026-03-25 的后续澄清收敛为“活动日期范围 + 用户本地优惠窗口”；Codex 第二行显示“当前持续优惠时段”。

### Session 2026-03-25

- Q: Claude Code 详情第二行的时间信息应如何表达？ → A: 显示“活动日期范围 + 用户本地优惠窗口”，而不是只显示本地起止时间戳或只显示日期范围。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 一眼判断当前是否处于优惠状态 (Priority: P1)

用户打开主面板后，希望在不进入设置、不阅读长文案的前提下，立刻知道各个服务“现在是不是优惠时段”，并在需要时看到更完整的优惠结论与幅度，从而比较后决定优先使用哪个服务。

**Why this priority**: 这项能力直接服务于 PRD 的核心目标“零打断感知余量”。如果用户仍需要自己换算时区、查帮助中心或猜活动是否生效，功能就没有成立。

**Independent Test**: 打开主面板，仅查看标题下方的小胶囊提示与其对应完整浮层，用户应能在 1 秒内判断各个已支持服务当前是否存在可用优惠，并据此做出服务选择。

**Acceptance Scenarios**:

1. **Given** 当前时间命中某个服务公开定义的优惠时间窗口，**When** 用户打开主面板，**Then** 标题下方会明确显示该服务“正在优惠时段”。
2. **Given** 某个服务存在公开活动但当前时间不在其优惠时间窗口内，**When** 用户打开主面板，**Then** 标题下方会明确显示该服务“不在优惠时段”。
3. **Given** 当前 Codex 活动按产品语义被视为全天连续的优惠时段，**When** 用户打开完整浮层，**Then** 该服务显示为“正在优惠时段 2x”。
4. **Given** 多个服务同时存在活动，**When** 用户打开主面板，**Then** 标题下方应以一行紧凑的小胶囊并列方式优先表达当前有优惠或待确认的服务状态，帮助用户比较后选择服务。
5. **Given** 用户需要查看无优惠服务的完整对比，**When** 用户悬停、聚焦或点击标题下方的促销信息行，**Then** 标题下方同一块提示区域会锚定展开全部服务状态，而不是跳到新的独立区域，且每个服务以两行详情块展示。
6. **Given** 用户通过 click 打开了完整浮层，**When** 用户点击浮层外区域或按 `Esc`，**Then** 浮层应关闭并回到默认胶囊视图。
7. **Given** 当前没有任何已知且仍有效的优惠活动，**When** 用户打开主面板，**Then** 标题下方显示“当前无优惠活动”。

---

### User Story 2 - 活动信息可独立维护并保留历史 (Priority: P1)

产品或运营在未来新增、调整、结束优惠活动时，希望只更新“优惠活动”这一个独立产品域，而不需要改动已有额度产品定义；同时已结束活动仍然保留记录，便于回溯曾经展示过什么、何时展示、依据什么来源。

**Why this priority**: 优惠活动天然高频变化，且不同服务商表达方式不同。如果没有独立模块和历史保留，功能会在第二次活动变更时迅速失控。

**Independent Test**: 在不改变主面板其他信息结构的前提下，新增一个活动、结束一个活动、归档一个历史活动，均不会影响对现行活动的识别和展示。

**Acceptance Scenarios**:

1. **Given** 新的优惠活动开始，**When** 产品更新活动目录，**Then** 新活动可被纳入判断而不需要改变已有历史活动记录。
2. **Given** 某优惠活动结束，**When** 活动状态变为已结束，**Then** 主面板不再把它作为当前活动展示，但历史记录仍可保留其名称、服务、时间范围和来源。
3. **Given** 同一服务未来再次发起新活动，**When** 新活动上线，**Then** 系统将其作为新的活动实体处理，而不是覆盖旧活动历史。
4. **Given** 某活动的公开说明后续被服务方修订，**When** 产品更新活动定义，**Then** 变更后的活动规则生效，同时保留原规则版本的历史痕迹。
5. **Given** 后续维护者需要更新促销活动，**When** 其查看项目文档，**Then** 能找到一份明确的促销活动更新指南，了解如何新增、结束、归档和校验活动。

---

### User Story 3 - 面对多服务与不完整信息时不误导用户 (Priority: P2)

当多个服务同时存在活动、或者活动公开信息并不完整时，用户仍希望看到简短但可信的一行聚合提示，而不是混乱、过长或带有错误确定性的文案。

**Why this priority**: 这个场景决定了功能是否真正可用。促销信息本身比额度信息更容易含糊，如果产品用错误的确定性表达，会直接伤害用户信任。

**Independent Test**: 在“Claude Code 有明确时段活动 + Codex 按连续优惠时段处理 + 用户计划未知/部分不适用”的组合下，主面板仍能同时表达两个服务的当前状态；Codex 在完整浮层中显示“正在优惠时段 2x”，且不会把未知资格服务隐藏或误判。

**Acceptance Scenarios**:

1. **Given** 多个服务同时存在活动，且其中一个服务当前明确处于优惠时间窗口，**When** 用户打开主面板，**Then** 一行聚合提示会优先显示所有“有优惠/待确认”的服务状态，并让处于优惠时段的服务状态清晰可见。
2. **Given** 多个服务同时存在活动，但只有部分服务存在优惠，**When** 用户打开主面板，**Then** 默认提示不必包含无优惠服务，但用户仍可通过促销信息行展开查看全部服务状态。
3. **Given** 某活动的适用计划不覆盖当前用户所用计划，**When** 用户打开主面板，**Then** 该活动不应被判断为当前对用户有效的优惠提示。
4. **Given** 用户当前计划未知、缺失或无法可靠判断，**When** 用户打开主面板，**Then** 该服务仍应出现在聚合提示中，但状态必须显示为“待确认”或“未知”，而不是被隐藏或误判为生效。
5. **Given** 某活动的时间规则需要跨时区换算，**When** 用户打开主面板，**Then** 展示结果应与用户本地时间下的真实命中情况一致。

---

### Edge Cases

- 同一服务同时存在一个“全天有效的限时活动”和一个“仅特定时段生效的加成活动”时，该服务的展示结果优先体现时效性更强、行动价值更高的“当前是否处于优惠时段”判断。
- 对当前 Codex 活动，产品语义允许将其视为连续的优惠时段，并在完整浮层中带出 `2x` 幅度。
- Claude Code 在完整浮层中的第二行时间信息必须展示为“活动日期范围 + 用户本地优惠窗口”，而不是只显示原始来源时区规则或只显示本地起止时间戳。
- Codex 在完整浮层中的第二行应使用“当前持续优惠时段”这类语义化描述，不使用 `00:00-23:59` 这种机械时间串。
- 活动规则使用服务方时区且换算到用户本地后跨天时，仍需按用户当前本地时间判断是否命中。
- 活动仅适用于部分订阅计划时，未覆盖的计划不应显示为命中活动。
- 无法确认用户当前所属计划时，不得输出“正在优惠时段”这类确定性提示，应以“优惠资格待确认”状态继续显示该服务，而不是隐藏。
- 当默认视图采用“只显示有优惠或待确认的服务”时，仍必须提供一个可进入的“查看全部”视图，以避免用户误以为其他服务不存在。
- “查看全部”不应通过显式高权重按钮进入，也不应再额外占用一枚 `...` 入口来争夺头部注意力。
- “查看全部”展开后，内容仍必须留在标题下方同一提示区域内，不得跳到新的独立信息区块。
- 促销状态不能只靠颜色表达；即使采用胶囊颜色区分，也必须同时保留服务图标与状态短文案。
- 促销信息的完整视图不能只依赖 hover；必须同时支持 focus 或 click 的等效触发，避免鼠标移入移出造成抖动或对键盘用户不可达。
- 当用户通过 click 稳定展开完整浮层后，必须存在清晰且一致的关闭方式；点击外部区域与按 `Esc` 都应关闭浮层。
- 历史活动与当前活动名称相似或相同，也必须作为不同活动记录管理。
- 如果所有活动信息都失效、撤回或无法确认，主面板必须回到“当前无优惠活动”，而不是继续沿用过期提示。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 产品必须引入独立的“优惠活动”产品域，用于管理活动定义、适用服务、活动状态、来源与历史记录；该域不得与额度展示规则混为同一概念。
- **FR-002**: 每个优惠活动必须具备可追溯的基础信息，至少包括：活动名称、归属服务、适用计划、活动类型、开始时间、结束时间或结束状态、信息来源、当前生命周期状态。
- **FR-003**: 产品必须支持至少两类优惠活动表达：`优惠时段型`（能判断当前时间是否命中）与 `限时活动型`（仅能判断活动当前有效，但无可公开判断的日内时段）。
- **FR-004**: 主界面标题下方必须保留一条小字号的优惠提示区域，用于展示当前值得用户关注的优惠状态，帮助用户比较后选择服务。
- **FR-004a**: 该优惠提示区域必须以聚合方式同时表达多个服务的状态，不得仅展示单一服务结果。
- **FR-004b**: 该优惠提示区域必须保持为一行紧凑表达，在不扩展为说明段落的前提下完成多服务状态传达。
- **FR-004c**: 默认视图中的服务状态应以“服务图标 + 极短状态文案”的并列小胶囊表达，而不是纯文本串接。
- **FR-004d**: 该优惠提示区域不得额外显示固定可见前缀“促销信息：”；应优先把横向空间留给服务状态本身。
- **FR-005**: 只有当活动规则公开且当前时间可被准确判断为命中时，产品才可以展示“正在优惠时段”。
- **FR-006**: 当活动当前有效但无法准确判断具体时段时，产品必须使用保守状态，不得冒充“正在优惠时段”。
- **FR-007**: 当没有任何当前有效且适用于用户的活动时，产品必须展示统一的否定态文案，明确告知当前无优惠活动。
- **FR-008**: 当多个服务同时存在活动时，产品必须为相关服务分别给出当前优惠状态，避免把多服务信息压缩为单一结论。
- **FR-008a**: 当同一服务下存在多个可同时命中的活动时，产品必须按稳定规则生成该服务的单一展示结论，优先级依次为：当前命中的优惠时段型活动、当前有效的限时活动、无活动。
- **FR-008b**: 默认视图必须优先显示当前“正在优惠时段”或“优惠资格待确认”的服务状态，以提高可扫读性。
- **FR-008c**: 产品必须提供一个“查看全部”能力，使用户可以查看所有已支持服务的当前状态，包括无优惠服务。
- **FR-008d**: “查看全部”应通过标题下方促销信息行自身触发，默认支持 hover 预览，不再额外引入显式 `...` 入口。
- **FR-008e**: hover 不能是唯一触发方式；该能力必须同时支持 focus 或 click 的等效进入方式。
- **FR-008e1**: hover 只用于临时预览；click 进入的是稳定展开态，而不是与 hover 完全等价的瞬时预览。
- **FR-008f**: “查看全部”浮层不持久化；每次面板重新打开时必须回到默认聚焦视图，以保持首屏信息的可扫读性。
- **FR-008f1**: 当用户处于 click 打开的稳定展开态时，点击浮层外区域或按 `Esc` 必须关闭浮层并回到默认胶囊视图。
- **FR-008g**: 用户进入“查看全部”状态后，全部服务状态必须仍锚定在标题下方同一提示区域内展开，而不是跳转到新的独立展示区域。
- **FR-008g1**: 完整浮层中的每个服务必须使用“两行信息块”展示，而不是压缩成单行长文本。
- **FR-008g2**: 两行信息块的第一行必须展示“图标 + 服务名 + 结论 + 可选优惠幅度”。
- **FR-008g3**: 两行信息块的第二行必须展示时间信息或时段说明；Claude Code 应展示“活动日期范围 + 用户本地优惠窗口”，Codex 应展示“当前持续优惠时段”。
- **FR-008h**: 当前版本的聚合表达应以现有已支持服务数量为边界进行优化；若未来服务数量增长导致同一区域内无法稳定承载紧凑表达，产品必须先通过缩写服务名、压缩状态短语等方式保持同一区域呈现，不得未经新一轮产品定义就自行扩展为新的独立信息区或长段落说明。
- **FR-009**: 优惠提示判断必须考虑活动适用计划；不适用于当前用户计划的活动不得进入当前提示。
- **FR-009a**: 当用户计划未知、缺失或无法可靠判断时，产品必须采用保守策略，不得把该活动判断为当前确定生效。
- **FR-009b**: 当用户计划未知、缺失或无法可靠判断时，该服务仍必须继续出现在优惠提示中，并以“优惠资格待确认”类中性状态展示。
- **FR-010**: 优惠提示判断必须以活动声明的官方规则为准，并换算为用户本地时间后再判断当前是否命中。
- **FR-011**: 活动结束后必须停止在主界面中作为当前活动展示，但其历史记录、曾用文案依据与生效区间必须保留。
- **FR-012**: 若活动规则被服务方更新，产品必须允许新增规则版本或修订记录，并确保后续判断基于最新有效规则。
- **FR-012a**: 该功能在实施后必须附带一份“促销活动更新指南”，用于指导后续维护者如何更新当前活动、保留历史活动、校验时间规则与验证展示结果。
- **FR-012b**: “促销活动更新指南”至少必须包含以下内容：活动目录位置与修改入口、关键字段说明、何时新增活动/结束活动/归档历史/追加规则版本、时间与时区校验步骤、默认胶囊视图与完整浮层视图的展示验证步骤，以及更新后应执行的测试或截图检查。
- **FR-013**: 当前版本必须纳入以下已知活动：
- **FR-013a**: Claude March 2026 usage promotion：2026 年 3 月 13 日至 2026 年 3 月 28 日 23:59 PT，有效对象为 Free、Pro、Max、Team 计划；工作日 8:00 AM-2:00 PM ET 之外属于加倍用量时段；覆盖 Claude Code。
- **FR-013b**: OpenAI Codex limited-time promotion：依据 OpenAI 帮助中心当前公开说明，Codex 目前存在限时活动；Free 与 Go 计划含 Codex，Plus、Pro、Business、Enterprise/Edu 享受 2x rate limits。当前产品版本将其展示语义定义为“连续优惠时段”，并在完整浮层中显示 `2x` 幅度。
- **FR-014**: 用户可见文案必须遵循“低认知负担、低误导风险”原则，避免出现需要用户自行换算时区或自行解释活动条件的表达。
- **FR-014a**: 用户可见状态语义不得只依赖颜色；颜色只能作为辅助编码，必须与图标和短文案共同出现。
- **FR-015**: 本功能的产品目标之一必须明确写为：通过展示所有服务的可用优惠信息，帮助用户选择当前更适合使用的服务。

### Key Entities *(include if feature involves data)*

- **PromotionCampaign**: 一个完整的优惠活动定义，描述某服务在某段时间内的优惠安排，包括活动名称、服务、计划覆盖范围、活动类型、官方来源和生命周期状态。
- **PromotionWindow**: 活动的时间规则。可表现为连续有效区间，也可表现为“仅工作日特定时段之外生效”这类重复性时间窗口。
- **PromotionEligibilityRule**: 活动适用条件，包括适用计划、适用产品表面，以及是否对当前用户有效。
- **PromotionCampaign (history)**: 活动的历史留档通过 `PromotionCampaign` 自身的 `lifecycle`（`ended` / `archived`）与 `historyNote` 字段承载，不拆分为独立实体；已结束活动保留在目录中，支撑将来回溯与审计。
- **PromotionDisplayDecision**: 某一时刻主界面针对每个服务生成的优惠提示结果集合，包含命中的活动、命中原因、展示文案类型与该服务内的优先级来源。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户打开主面板后，能够在 1 秒内通过标题下方的小胶囊提示判断当前值得关注的服务是否存在优惠时段或优惠活动。
- **SC-001a**: 在已支持服务数量范围内，优惠提示仍保持一行紧凑表达，不因多服务同时存在而退化为难以扫读的说明文字。
- **SC-002**: 在“正在优惠时段”“不在优惠时段”“无优惠活动”三类场景中，提示文案的误判率为 0。
- **SC-002a**: 在“用户计划未知”场景中，系统不会把服务误判为“正在优惠时段”或“明确不适用”，而是稳定展示“优惠资格待确认”。
- **SC-003**: 当前纳入的 Claude Code 与 Codex 两类活动可以同时存在且稳定输出多服务聚合提示；默认视图聚焦“正在优惠时段/优惠资格待确认”服务，查看全部视图可覆盖全部已支持服务，且不出现错误确定性表述。
- **SC-003a**: 用户可以在当前主界面中通过促销信息行的 hover/focus/click 发现并进入“查看全部”状态，而不会因额外入口过重影响头部信息扫读。
- **SC-003b**: 默认视图中的各服务状态可通过小胶囊的图标、短文案与辅助颜色在一眼内区分，不需要先阅读长串文本。
- **SC-003c**: 用户通过 click 打开完整浮层后，能在不猜测的情况下通过点击外部或按 `Esc` 关闭浮层，并稳定回到默认胶囊视图。
- **SC-003d**: 在真实壳层中，完整浮层的两行详情块仍保持可读，不因日期范围、时区说明或优惠幅度信息而出现明显重叠、跳动或难以扫描的问题。
- **SC-004**: 任一活动结束后，主界面在下一个有效判断周期内不再展示其为当前活动，同时历史活动记录完整保留。
- **SC-005**: 当活动规则发生修订时，产品可在不删除旧活动记录的前提下完成规则更新，并保持当前判断结果与最新公开规则一致。

## Scope

**In scope**:

- 主界面标题下方的单行优惠提示
- 标题下方一行紧凑的多服务胶囊聚合表达
- “默认聚焦有优惠服务 + 可预览全部”的展示模式
- 主界面内通过促销信息行 hover/focus/click 查看全部的交互
- 面向 Claude Code 与 Codex 的当前活动纳入
- 优惠活动独立产品域与历史记录保留
- 促销活动更新指南
- 多服务并存下的聚合展示规则
- 时区换算、计划适用性与公开信息完整度对提示文案的影响

**Out of scope**:

- 在卡片内部展开活动详情
- 活动订阅、活动通知、弹窗营销或推送
- 自动推断服务方未公开的优惠规则
- 为用户展示完整活动条款全文
- 将优惠活动作为额度计算逻辑的一部分对外解释

## Assumptions

- 该功能的首要价值是“帮助用户判断当下是否更值得使用某服务”，而不是完整承载营销说明。
- 主界面头部的提示区域虽然轻量，但仍需承担“展示所有服务当前优惠状态”的职责，以支持用户做服务选择。
- 当前用户可能同时使用多个服务，因此该提示需要服务间可比较，而不只是给出单一结论。
- 当前版本支持的服务数量有限，因此“一行紧凑表达所有服务状态”在产品层面可成立，并优先于扩展为多行说明。
- 当前版本按现有已支持服务数量进行优化；若未来服务继续增加，应先压缩文案和服务简称来维持同一区域表达，而不是在未重新做产品设计时自行扩展成新的区域或说明块。
- 为兼顾可扫读性与完整性，默认视图可以只显示“正在优惠时段/优惠资格待确认”的服务，但必须允许用户查看全部服务状态。
- 默认视图应以小胶囊承载服务状态，不显示固定“促销信息：”前缀，把空间优先留给可比较的信息本体。
- 完整浮层承担更高的信息密度，应按服务拆成两行信息块，而不是继续复用默认胶囊的极简表达。
- “查看全部”默认通过悬停在促销信息行上进行预览，同时必须保留 focus/click 的等效进入方式，避免 hover-only 带来的可发现性与稳定性问题。
- hover 用于快速预览，click 用于稳定展开；稳定展开后的退出路径统一为点击外部区域或按 `Esc`。
- Claude 官方页面公开了明确活动日期与工作日时段规则，因此可以判断“当前是否处于优惠时段”。
- Claude Code 完整浮层中的时间信息应同时包含活动日期范围和用户本地优惠窗口，而不是要求用户自己换算 ET/PT 规则。
- 当前产品版本将 Codex 已公开促销活动解释为“全天连续优惠时段”，因此默认可展示为“正在优惠时段”，并在完整浮层中补充 `2x` 幅度。
