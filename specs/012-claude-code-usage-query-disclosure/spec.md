## Clarifications

### Session 2026-03-24

- Q: 用户在设置页启用 Claude Code 用度查询后，首次查询应在何时发生？ → A: 立即触发首次查询，并展示真实结果或真实不可用状态。
- Q: Claude Code 用度查询关闭后，现有相关设置项应如何表现？ → A: 从现有相关设置项中隐藏 Claude Code，避免继续形成无效配置。
- Q: 新增界面文案的语言范围应如何定义？ → A: 首次引导卡片与设置页新增文案都需支持中英文切换；开关相关语义（含无障碍标签）也需随语言切换；README 沿用仓库现有语言策略。
- Q: Claude Code 是唯一启用的服务时关闭后，主面板和菜单栏应如何表现？ → A: 回退到"尚未连接任何服务"的空状态。
- Q: 关闭 Claude Code 用度查询时，已缓存的额度数据应如何处理？ → A: 保留缓存数据但不展示，重新启用时优先使用缓存再刷新。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 首次看到清晰可信的 Claude Code 告知 (Priority: P1)

首次安装或处于“尚未连接任何服务”的用户打开 AIUsage 时，会在现有首次引导界面里看到一张独立的 Claude Code 查询说明卡片。用户无需先翻文档或误触功能，就能知道应用会读取什么、不会做什么，以及 Claude Code 查询默认不会自动启用。

**Why this priority**: PRD 的核心目标是“零打断感知余量”和“本地优先、可审计”。对于涉及本机登录凭证和外部额度查询的能力，第一次看见时就说清楚边界，是建立用户信任的前提；同时用卡片承载而不是阻断式流程，也能保持轻量、低打断的产品体验。

**Independent Test**: 在满足首次引导显示条件的环境中打开应用，验证引导区出现独立的 Claude Code 说明卡片；点击“我知道了”后，仅该卡片消失而整个首次引导仍保留；再次回到同样的首次引导状态时，该卡片不再出现。

**Acceptance Scenarios**:

1. **Given** 用户第一次进入“尚未连接任何服务”的首次引导界面，且此前未确认过 Claude Code 说明，**When** 引导内容加载完成，**Then** 用户会看到一张独立的 Claude Code 查询说明卡片，并能识别 `Claude Code 查询` 标识。
2. **Given** 用户正在查看 Claude Code 说明卡片，**When** 用户点击 `我知道了`，**Then** 仅该卡片被关闭，首次引导本身不因此结束，也不会自动启用 Claude Code 用度查询。
3. **Given** 用户此前已经点击过 `我知道了`，**When** 之后再次进入相同的首次引导状态，**Then** Claude Code 说明卡片不再重复出现。

---

### User Story 2 - 在设置页显式决定是否启用 Claude Code 用度查询 (Priority: P1)

用户进入设置页后，能够在页面末尾看到一个独立的 Claude Code 查询卡片。卡片标题为 `Claude Code 查询`，右侧直接提供启用开关。用户默认处于关闭状态，可在充分了解用途边界后，自主决定是否开启，而无需经历额外确认流程；开启后系统会先进入明确可见的查询中状态，再收敛到真实结果。

**Why this priority**: PRD 强调“打开菜单栏弹窗 < 1 秒即可看到所有服务余量”，但这一前提不应建立在用户不知情的敏感行为之上。显式开关让产品同时满足“轻量无感”和“行为可控”，是把信任边界转化为产品能力的核心交付。

**Independent Test**: 打开设置页，验证原有设置顺序保持不变，Claude Code 相关内容作为与主设置卡片同级的独立卡片出现在最底部；验证卡片标题为 `Claude Code 查询`，右侧开关默认关闭，用户可直接开启或关闭，且重启后仍保持选择结果；开启后先看到明确的查询中状态，再看到真实结果或真实不可用状态。

**Acceptance Scenarios**:

1. **Given** 用户打开设置页，**When** 页面渲染完成，**Then** 用户会在所有现有设置项之后看到一个独立的 Claude Code 查询卡片，卡片标题为 `Claude Code 查询`，开关位于标题右侧。
2. **Given** 用户此前从未明确选择过 Claude Code 用度查询，**When** 用户第一次查看设置页中的该开关，**Then** 开关默认处于关闭状态。
3. **Given** 用户正在设置页操作 Claude Code 用度查询开关，**When** 用户切换开关，**Then** 该选择立即生效并被记住，且不会出现额外确认弹层或阻断页面。
4. **Given** 用户刚刚启用 Claude Code 用度查询，**When** 开关状态变为开启，**Then** 系统会立即进入一个明确可见的查询中状态，并同时发起一次 Claude Code 用度查询。
5. **Given** 用户已经启用 Claude Code 用度查询但首次查询尚未完成，**When** 用户查看主面板、菜单栏或设置页中的相关可见状态，**Then** 系统会持续展示真实的查询中状态，而不是把 Claude Code 误显示为可用、空白或静默无变化。
6. **Given** 用户刚刚启用 Claude Code 用度查询，**When** 首次查询完成，**Then** 查询中状态会被真实结果或真实不可用状态替换，不要求用户额外执行手动刷新或等待下一次自动刷新。
7. **Given** 用户此前已关闭 Claude Code 用度查询且本地仍保留有缓存数据，**When** 用户重新开启开关，**Then** 系统会立即展示缓存数据并伴随可见的刷新中指示，同时触发一次真实查询（受冷却间隔约束）；查询完成后，缓存数据被真实结果替换。
8. **Given** 用户刚刚关闭再立即重新开启 Claude Code 用度查询，且距上一次成功查询尚未超过最小冷却间隔，**When** 开关变为开启，**Then** 系统直接使用已有缓存呈现结果，跳过本次实际请求，不向用户暴露为错误。
9. **Given** 用户当前使用英文界面，**When** 用户查看 Claude Code 说明和开关区域，**Then** 新增文案会以英文完整显示，不出现中英混杂。

### User Story 3 - 关闭后系统行为真实一致，不再暗中触发 Claude Code 查询 (Priority: P1)

对敏感行为更谨慎的用户在关闭 Claude Code 用度查询后，能够确信 AIUsage 不会继续把 Claude Code 当作启用中的服务，不会继续读取本机 Claude Code 登录凭证，也不会继续发起 Claude Code 额度查询。界面和实际行为保持一致，不出现“明明已关闭，却仍像启用中”的矛盾状态。

**Why this priority**: 这直接对应 PRD 的“轻量无感”和“本地优先、可审计”原则。对用户来说，明确的关闭必须意味着明确的停止；否则任何告知文案都会失去可信度。

**Independent Test**: 在 Claude Code 曾经可用或已被选为当前展示服务的环境中关闭该开关，验证 Claude Code 不再被展示为当前有效服务，之后的自动刷新、手动刷新和重启也都不会恢复 Claude Code 用度查询，直到用户再次明确开启。

**Acceptance Scenarios**:

1. **Given** 用户已关闭 Claude Code 用度查询，**When** 应用发生初始化、自动刷新或手动刷新，**Then** 应用不会再读取 Claude Code 登录凭证，也不会再发起 Claude Code 额度查询。
2. **Given** 用户关闭 Claude Code 用度查询前，Claude Code 正被作为当前菜单栏展示服务，**When** 关闭操作生效，**Then** 当前展示会立即回退到仍然有效的其他服务，不保留 Claude Code 的激活态表现。
3. **Given** 用户已关闭 Claude Code 用度查询，**When** 用户查看主面板和设置结果，**Then** 不会看到 Claude Code 仍被当作启用中或正在刷新中的矛盾表现。
4. **Given** 用户已关闭 Claude Code 用度查询，**When** 用户查看仍然存在的相关设置项，**Then** 这些设置项不再提供 Claude Code 作为可选项，也不再展示 Claude Code 的排序入口。

---

### User Story 4 - 在 README 中提前看到行为边界说明 (Priority: P2)

准备安装、评估或审计 AIUsage 的用户，可以在 README 中提前看到 Claude Code 用度查询的行为边界说明，而不必先启动应用或翻查代码，便能理解这一能力的用途和边界。

**Why this priority**: PRD 明确强调“开源，用户可自行审计”。把这类边界说明放进 README，能把“可审计”从源代码层面前移到产品沟通层面，降低理解成本，也更符合开源产品的透明预期。

**Independent Test**: 打开 README，验证其中存在独立的 Claude Code 查询说明小节，且小节覆盖本机凭证用途、非官方目标边界和代理提示三类关键信息。

**Acceptance Scenarios**:

1. **Given** 用户在仓库首页阅读 README，**When** 用户浏览功能或使用说明内容，**Then** 用户能够看到一个独立的 Claude Code 查询说明小节。
2. **Given** 用户阅读该小节，**When** 用户查看说明正文，**Then** 用户能够明确知道 Claude Code 登录凭证仅用于向 Claude 官方接口查询额度状态，不会被存储、修改或主动发送到 AIUsage 自己的服务或其他非官方接口。

---

### Edge Cases

- 老用户从早期版本升级后，即使本机已经存在可用的 Claude Code 登录凭证，也必须先处于“未启用”状态，直到用户主动打开开关。
- 用户关闭 Claude Code 说明卡片，只代表“我已阅读此说明”，不代表“我已启用 Claude Code 用度查询”，也不影响首次引导本身的关闭逻辑。
- 用户可能已经跳过或关闭过首次引导，因此即使不再出现首次引导卡片，设置页和 README 仍必须完整承担 Claude Code 告知职责。
- 用户开启 Claude Code 用度查询后，若本机没有有效登录凭证、网络不可达或代理不满足要求，系统仍应展示真实的不可用或待处理状态，而不是把“已开启”误导成“已成功连接”。
- 用户开启 Claude Code 用度查询后，若首次查询耗时较长，系统必须保持明确的查询中状态，直到成功返回真实额度数据或明确进入真实不可用状态，不能在中间退回空白、旧缓存冒充最新结果或“已开启但无状态”的静默表现。
- 用户关闭 Claude Code 用度查询时，如果界面上仍保存了与 Claude Code 相关的当前展示或排序结果，系统必须在同一次设置生效周期内完成归一化，避免留下无效状态。
- 用户关闭 Claude Code 用度查询后，任何仍然存在的服务选择或排序入口都不应继续暴露 Claude Code，否则会重新制造”已关闭但仍可配置”的矛盾感。
- 当 Claude Code 是用户唯一启用的服务且被关闭时，系统必须回退到”尚未连接任何服务”的空状态，而不是保留空白面板或显示无效的 Claude Code 残留。
- 用户在短时间内反复开关 Claude Code 用度查询时，所有实际官方接口请求必须受同一个冷却机制约束，冷却期内的请求静默跳过并复用缓存，避免触发 429 速率限制。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须在 README 中提供一个独立的 Claude Code 查询说明小节，作为长期可见的公开说明入口。
- **FR-002**: README 说明必须明确表达以下边界：AIUsage 会读取本机现有的 Claude Code 登录凭证；该凭证仅用于向 Claude 官方接口查询额度状态；AIUsage 不会存储、修改或管理该凭证，也不会主动将其发送到 AIUsage 自己的服务或其他非官方接口；部分地区可能需要网络代理，程序会自动检测并使用可用代理配置。
- **FR-003**: 当用户进入现有首次引导界面且尚未确认过 Claude Code 说明时，系统必须在引导界面中展示一张独立的 Claude Code 查询说明卡片。
- **FR-004**: 首次引导中的 Claude Code 说明卡片必须包含清晰、简短且与 README 核心边界一致的说明内容、醒目的 `Claude Code 查询` 标识，以及一个明确的确认动作 `我知道了`。
- **FR-005**: 用户点击 `我知道了` 后，系统必须只关闭这张 Claude Code 说明卡片，并持久化该确认结果，使其之后不再重复出现。
- **FR-006**: 确认 Claude Code 说明卡片不得自动启用 Claude Code 用度查询，也不得替代现有首次引导的整体关闭或跳过逻辑。
- **FR-007**: 系统必须在设置页最底部新增一个与主设置卡片同级的 Claude Code 独立卡片；卡片标题为 `Claude Code 查询`，启用开关位于标题右侧，现有设置项的相对顺序必须保持不变。
- **FR-008**: 设置页中的 Claude Code 说明必须与首次引导和 README 传达同一组行为边界，确保用户无论从哪个入口进入，都能理解该能力的用途和限制。
- **FR-008a**: 首次引导卡片与设置页中的新增说明文案必须跟随应用当前语言在中文与英文之间切换；启用开关相关语义（含无障碍标签）也必须同步切换，避免出现中英混杂或仅部分翻译的状态。
- **FR-009**: Claude Code 用度查询必须对所有尚未明确选择的用户默认关闭，包括新用户和从旧版本升级的现有用户。
- **FR-010**: 用户必须能够在设置页直接启用或关闭 Claude Code 用度查询，系统必须立即生效并记住用户选择，且整个过程不得弹出额外确认弹层。
- **FR-010a**: 当用户启用 Claude Code 用度查询时，系统必须立即进入可见的查询中状态，并同时触发一次 Claude Code 用度查询，不得要求用户额外执行手动刷新或等待下一次自动刷新才看到首次结果。若距上一次成功查询（无论来源是自动刷新、手动刷新还是开关触发）尚未超过最小冷却间隔，则跳过本次实际请求，直接使用已有缓存呈现结果。
- **FR-010b**: 当 Claude Code 首次查询尚未完成时，主面板、菜单栏和设置页中与 Claude Code 相关的可见状态必须忠实展示”查询中”或等价的 loading 语义，不得把 Claude Code 表现为已可用、空白无状态或静默失败。
- **FR-010c**: 所有 Claude Code 官方接口调用（开关触发、手动刷新、自动刷新）必须共享同一个请求冷却机制，确保任意两次实际请求之间不低于最小冷却间隔，以避免触发 429 速率限制。冷却期内的请求应静默跳过并复用已有缓存，不向用户暴露为错误。
- **FR-011**: 当 Claude Code 用度查询处于关闭状态时，系统不得读取 Claude Code 登录凭证，不得发起 Claude Code 官方额度查询，也不得在界面上把 Claude Code 表现为正在刷新、已启用或可用中的服务。已缓存的 Claude Code 额度数据应保留但不展示。
- **FR-012**: 当 Claude Code 用度查询处于开启状态时，系统必须恢复现有的 Claude Code 额度查询能力，优先使用已有缓存数据立即呈现，并同时触发刷新以获取最新状态。当展示缓存数据时，必须伴随可见的刷新中指示，不得将旧缓存伪装为当前最新结果。
- **FR-013**: 当用户关闭 Claude Code 用度查询且当前界面仍指向 Claude Code 相关展示时，系统必须在同一次设置变更中立即回退到仍然有效的展示结果，避免保留无效激活态。若 Claude Code 是唯一启用的服务，则回退到"尚未连接任何服务"的空状态。
- **FR-013a**: 当 Claude Code 用度查询处于关闭状态时，所有仍然存在的相关设置项都不得继续提供 Claude Code 作为可选项或排序项。
- **FR-014**: 即使用户从未进入首次引导，系统也必须确保设置页和 README 足以独立完成 Claude Code 用度查询的用户告知。

### Key Entities *(include if feature involves data)*

- **Claude Code 用度查询偏好**: 表示用户是否明确允许 AIUsage 使用本机现有的 Claude Code 登录凭证来查询官方额度状态；该偏好决定 Claude Code 相关查询链路是否可以生效。
- **Claude Code 告知确认状态**: 表示用户是否已经阅读并确认过首次引导中的 Claude Code 说明卡片；该状态只控制说明卡片是否再次出现，不代表功能启用。
- **Claude Code 告知内容**: 指 README、首次引导卡片和设置页中需要保持语义一致的说明文本，用于向用户传达凭证用途边界、非官方目标边界和代理提示。

### Assumptions

- 本迭代复用现有首次引导界面承载 Claude Code 说明，不新增阻断式确认流程。
- Claude Code 用度查询是一项可选能力，而不是应用默认必须启用的基础能力。
- 当 Claude Code 被关闭后，仍然有效的其他服务可以承担菜单栏或主面板的默认展示角色。
- 设置页始终可被用户访问，因此即使用户错过首次引导，也仍有机会了解并控制 Claude Code 用度查询。
- README 的语言呈现方式沿用仓库现有文档策略，不要求本迭代额外扩展新的 README 语言体系。

### 可信边界声明

- 本功能的产品目标不是“替用户管理 Claude Code 账号”，而是让用户在知情和可控的前提下，选择是否允许 AIUsage 使用本机现有登录凭证查询官方额度状态。
- 告知内容必须优先建立“本地优先、不开云同步、可由用户审计”的信任感，而不是把敏感行为隐藏在异常文案、错误提示或实现细节中。
- 启用控制的设计必须同时兼顾 PRD 的“轻量无感”目标：用户应能在一处看懂边界、在一处做出选择，而不是被反复确认或被迫中断主流程。

### Non-Goals

- 本迭代不重新设计整个首次引导页面，只在现有首次引导中新增一张独立的 Claude Code 说明卡片。
- 本迭代不重排现有设置项的顺序，只在设置页最底部追加独立的 Claude Code 查询卡片。
- 本迭代不扩展为“所有服务统一敏感行为告知框架”；范围仅覆盖 Claude Code 用度查询。
- 本迭代不改造代理配置流程本身，只要求在说明中清楚告知代理对 Claude Code 用度查询的影响。
- 本迭代不引入云端同步、账号托管、凭证代管或新的账户管理能力。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在满足首次引导显示条件且用户尚未确认过 Claude Code 说明的情况下，100% 会展示独立的 Claude Code 说明卡片。
- **SC-002**: 用户点击 `我知道了` 后，仅 Claude Code 说明卡片消失，且该卡片在后续相同首次引导场景中不再重复出现。
- **SC-003**: 设置页保留原有设置顺序不变，Claude Code 查询卡片始终位于最后一个独立卡片位置，且与主设置卡片同级。
